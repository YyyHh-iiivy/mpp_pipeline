# Week 2 大核媒体管线接口说明

更新时间：2026-06-08

本文档给队员 A/B 对接使用。当前代码仍保持 Week 1 主链路：

```text
GC2093 Sensor -> VICAP/VI -> kd_mpi_sys_bind -> VENC(H.265) -> stream_thread
```

大核只提供稳定码流、事件和控制接口。RTSP、RTP/H.265 打包和网络发送由小核 Linux 侧实现。

## 1. 大小核职责边界

大核 RT-Smart：

- 管理 VB / VICAP / VI / VENC。
- 通过 `kd_mpi_sys_bind` 保持 1080P 主链路零拷贝。
- 从 VENC 获取 H.265 pack 描述。
- 后续负责 OSD Region 和低分辨率 AI 旁路。
- 运动事件优先在大核闭环触发 OSD，保护 300ms 延迟目标。

小核 Linux：

- 运行 RTSP server。
- 完成 RTP/H.265 打包与网络发送。
- 管理 Wi-Fi / Ethernet / VLC 或 ffplay 拉流入口。
- 通过官方 MAPI/IPC/共享内存消费大核提供的数据。

跨核边界：

- 优先传控制消息、共享内存句柄、物理地址或官方支持的数据结构。
- 不传普通虚拟指针。
- 小核消费 VENC buffer 后必须有释放/归还机制，不能永久占用编码器 buffer。

## 2. H.265 码流出口字段

当前大核出口类型定义在 `mpp_types.h`：

```c
k_s32 stream_export_init(stream_export_mode mode);
k_s32 stream_export_submit(const mpp_stream_frame_desc *frame);
void stream_export_deinit(void);
```

`mpp_stream_frame_desc` 字段：

| 字段 | 含义 |
| --- | --- |
| `chn` | VENC 通道 ID，当前为 `VENC_CHN` / `0`。 |
| `pts` | 编码时间戳，来自 SDK `k_venc_pack.pts`。 |
| `pack_cnt` | 本帧 pack 数量，上限 `MPP_MAX_STREAM_PACKS` / `8`。 |
| `packs[].phys_addr` | pack 物理地址，来自 SDK `k_venc_pack.phys_addr`；后续跨核共享优先使用该字段。 |
| `packs[].virt_addr` | 当前固定为 `0`；不能跨核传普通虚拟地址。 |
| `packs[].len` | pack 长度，来自 SDK `k_venc_pack.len`。 |
| `packs[].type` | `K_VENC_HEADER`、`K_VENC_I_FRAME`、`K_VENC_P_FRAME` 等 SDK pack 类型。 |

当前 `stream_export_submit()` 的模式是 `STREAM_EXPORT_LOCAL_LOG`，只打印：

```text
[MPP] Get NALU, Size: xxxx bytes  [frame #N]
[MPP] Get NALU (SPS/PPS header), Size: xxxx bytes
```

后续接小核 RTSP 时，优先只替换 `stream_export_submit()` 内部实现，不让 `venc_module.c` 直接写网络或 RTSP。

## 3. 运动事件消息字段

当前预留类型：

```c
typedef struct {
    k_u32 event_id;
    k_u64 detect_time_ms;
    k_u32 motion_score;
    k_u32 osd_duration_ms;
    k_u32 request_snapshot;
} motion_event_msg;
```

字段约定：

- `event_id`：运动事件递增 ID。
- `detect_time_ms`：检测到运动的时间戳，单位 ms。
- `motion_score`：队员 B 帧差法输出分数。
- `osd_duration_ms`：OSD 显示时长，推荐 1000ms。
- `request_snapshot`：非 0 表示请求快照。

## 4. 快照请求字段

当前预留类型：

```c
typedef struct {
    k_u32 event_id;
    k_u64 frame_time_ms;
    k_u32 source_chn;
    char path_hint[64];
} snapshot_request_msg;
```

字段约定：

- `event_id`：对应运动事件 ID。
- `frame_time_ms`：快照参考帧时间戳。
- `source_chn`：快照来源通道，主码流为 `VENC_CHN`。
- `path_hint`：小核或存储线程可使用的路径提示，不传裸指针。

## 5. OSD 控制函数

当前接口：

```c
k_s32 osd_init(void);
k_s32 osd_set_motion_visible(k_u32 visible, k_u32 duration_ms);
void osd_deinit(void);
```

当前实现状态是 stub，会打印：

```text
[MPP] OSD init stub: waiting for VENC 2D OSD MMZ buffer wiring
[MPP] OSD motion visible=1 duration=1000ms (stub)
```

SDK 已确认存在 VENC 2D OSD API：

```c
k_s32 kd_mpi_venc_set_2d_osd_param(k_u32 chn_num, k_u8 index, const k_venc_2d_osd_attr *attr);
```

真实硬件 OSD 后续需要单独补齐 MMZ 图像 buffer、cache flush、alpha 控制和板端画面验证。当前 stub 不能视为硬件 OSD 已完成。

## 6. 低分辨率 AI 帧字段

推荐方案：

- 使用 VICAP/VI 第二通道做低分辨率旁路，不反压 1080P 主链路。
- 目标分辨率：640x480。
- 像素格式：NV12。
- 队员 B 帧差法只处理 Y 分量。
- 帧率：5 到 15fps。
- AI 处理跟不上时丢旧帧、保留最新帧，不阻塞主码流。
- 获取帧的一方负责调用对应 SDK release API 释放或归还 buffer。

后续接口草案：

```c
k_s32 ai_frame_channel_init(void);
k_s32 ai_frame_try_get(/* output frame desc */);
k_s32 ai_frame_release(/* frame handle */);
void ai_frame_channel_deinit(void);
```

当前未实现 `ai_frame_module.c`，原因是 VICAP 第二通道与释放 API 需要上板确认，不能猜 SDK 调用。

## 7. 禁止事项

- 不在大核实现 RTSP 或 RTP 主路线。
- 不用 V4L2、ffmpeg、gstreamer。
- 不 `memcpy` 1080P 原始帧给小核。
- 不传普通虚拟指针跨核。
- 不在 1080P 像素数组里 CPU 循环画 OSD。
- 不把 `SConscript` 改成 `Glob('*.c')`。

## 8. Week 2 对接顺序

1. 队长保持大核主链路稳定，确认 `stream_thread()` 无永久阻塞和每帧动态分配。
2. 队员 A 先跑通小核 Linux 官方 RTSP demo，找到 H.265 数据入口。
3. 队长和队员 A 对齐 `chn`、`pts`、`pack_cnt`、`phys_addr`、`len`、`type`、release 机制。
4. 队员 B 先完成输入 Y 分量、输出 `motion_score` 的帧差法。
5. 队长后续把低分辨率帧和 OSD 真实硬件实现接入事件链路。
