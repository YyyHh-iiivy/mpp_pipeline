# 运动检测灵敏度优化后 4 秒延迟重新诊断

## 1. 诊断范围

本次重新分析以以下内容为准：

- 实际测试代码目录：`mpp_pipeline--/mpp_pipeline--`
- 大核代码：`big_core/mpp_pipeline/`
- 用户上传的 `big_app_new.elf` 运行日志
- 目录中的 `程序流程.md`、`SNAPSHOT_BIG_LITTLE_GUIDE.md` 等设计文档

本次只进行代码和日志诊断，没有修改运行代码。

## 2. 核心结论

当前证据不支持“画面变化越大，运动检测算法计算越慢，直接把主码流拖到 4 秒”这一判断。

更可能的实际链路是：

```text
运动检测灵敏度提高
-> 检出的运动事件数量增加
-> 每个事件都申请快照
-> 每次快照都要求主 VENC 立即产生 IDR
-> IDR 瞬时数据量和小核快照处理量增加
-> 小核 RTSP/RTP、快照线程或播放器队列受到影响
-> 端到端延迟逐渐累积到约 4 秒
```

本次日志显示大核主 VENC 和 DATAFIFO 在正常运行阶段保持稳定，因此如果测试时确实观察到约 4 秒延迟，积压位置更可能位于小核读取 DATAFIFO 之后，或者播放器内部，而不是大核运动检测函数本身。

## 3. 为什么不是“大画面变化让检测算法突然算得很慢”

### 3.1 当前算法的每帧计算量基本固定

`big_core/mpp_pipeline/motion_detect.c` 中的强符号运动检测实现，每帧主要执行：

1. 第一遍按照 `MOTION_SAMPLE_STEP=4` 采样，计算全局亮度偏移。
2. 第二遍使用相同采样位置，建立补偿差值直方图。
3. 遍历固定的 256 个直方图区间。
4. 把当前 `640x480` 的 Y 平面复制为下一帧参考图像。

对于 `640x480` 图像，每一遍采样点数量为：

```text
(640 / 4) x (480 / 4) = 160 x 120 = 19200 个采样点
```

因此每帧固定执行两遍约 19200 点采样，再复制 307200 字节参考帧。

画面中变化像素变多，只会改变 `changed_pixels` 和 `motion_score` 的数值，不会增加循环次数。该算法不存在“运动区域越大，遍历次数越多”的数据相关复杂度。

### 3.2 日志证明运动检测线程没有明显变慢

日志停止时输出：

```text
[MPP] AI motion thread stopped, frames=716
```

同一轮主码流统计的运行时间约为 25 秒，因此：

```text
716 / 25 = 28.64 FPS
```

这说明运动检测线程仍然接近传感器的 30fps 运行，没有出现算法只能处理几帧每秒的情况。

如果运动检测函数真的因为画面变化变得非常慢，AI 线程帧数应明显下降，而不是保持约 28.6fps。

## 4. 主 VENC 和大核 DATAFIFO 日志分析

### 4.1 主 VENC 帧率稳定

正常运行期间连续出现：

```text
[MPP] === Stats: 150 frames in 10.0s, 15.0 fps, ... pending=1 ===
[MPP] === Stats: 150 frames in 10.0s, 15.0 fps, ... pending=1 ===
```

说明主 VENC 连续两个统计周期都保持 15fps，没有因运动检测出现零包停顿或明显降帧。

### 4.2 DATAFIFO 没有形成 4 秒积压

关键统计包括：

```text
seq=150 pending=1 datafifo_full=0 pending_full=0 write_fail=0
seq=300 pending=1 datafifo_full=0 pending_full=0 write_fail=0
```

对应的 `READ_DONE` 延迟约为：

```text
read_done_age_ms=67
```

这表明正常运行期间：

```text
主 VENC
-> 大核 stream_thread
-> DATAFIFO
-> 小核读取并 READ_DONE
```

没有出现持续积压。

日志末尾的：

```text
[MPP] NALU IPC READ_DONE: seq=386 read_done_age_ms=1244 ...
```

发生在 `Ctrl+C` 后的清理阶段，VICAP、stream thread 和 DATAFIFO 正在停止，不能用它判断正常运行期延迟。

### 4.3 当前日志没有体现 4 秒延迟发生在大核

本轮日志中没有出现以下运行期异常：

- 主 VENC 长时间没有输出。
- VENC 帧率从 15fps 大幅下降。
- `pending` 长期达到 3。
- `datafifo_full` 增长。
- `pending_full` 增长。
- `write_fail` 增长。
- `READ_DONE` 达到数秒。

因此不能把观察到的 4 秒端到端延迟归因于大核 VENC 或大核 DATAFIFO 队列。

## 5. 当前最可疑的调用链：运动事件触发快照和强制 IDR

### 5.1 每次运动事件都申请快照

`big_core/mpp_pipeline/motion_adapter.c` 中：

```c
event->request_snapshot = 1;
```

这意味着每个通过连续帧确认、锁存和冷却判断的运动事件都会申请快照。

### 5.2 AI 线程把快照请求交给码流导出层

`big_core/mpp_pipeline/motion_thread.c` 中：

```c
if (event.request_snapshot) {
    ...
    ret = stream_export_request_snapshot(&snapshot_req);
}
```

### 5.3 快照请求立即强制主 VENC 产生 IDR

`big_core/mpp_pipeline/stream_export.c` 中：

```c
idr_ret = kd_mpi_venc_request_idr(req.source_chn);
```

因此运动检测虽然没有直接控制主码流计算，但它通过事件链间接改变了主 VENC 的编码行为。

### 5.4 日志中已经体现该链路

本次约 25 秒的日志中出现了 5 次运动事件，每次事件都紧跟：

```text
Motion detected
-> Snapshot request queued
-> Snapshot immediate IDR requested
-> snapshot flag set
-> Snapshot request delivered to DATAFIFO
```

这说明当前运动检测和主码流之间并非完全隔离。灵敏度提高后，运动事件更容易成立，强制 IDR 和小核快照处理也会更频繁发生。

## 6. 强制 IDR 和小核快照为什么可能造成延迟

### 6.1 IDR 帧瞬时数据量较大

大画面突变时，H.265 编码器本来就需要描述更多变化。此时再强制生成 IDR，瞬时码流尺寸可能明显大于普通 P 帧。

当前虽然使用 CBR，但 CBR主要约束一段时间内的码率，不代表每一帧大小完全固定。因此需要统计每个 IDR 的总字节数，而不能只看十秒平均码率。

### 6.2 小核可能需要复制整张快照码流

设计文档要求小核在发送 `DATAFIFO_CMD_READ_DONE` 前复制带快照标志的编码数据，然后交给快照线程。

如果实际实现中存在以下情况，就可能阻塞或抢占流媒体任务：

- 在 DATAFIFO/RTP 线程中直接写 SD 卡。
- 在 DATAFIFO/RTP 线程中调用 `ffmpeg`。
- 快照线程与 RTSP/RTP 线程同优先级并大量占用 CPU。
- 快照请求队列无界增长。
- 同时运行多个 `ffmpeg` 转换任务。
- RTSP 发送队列只追加旧帧，不在拥塞时丢弃。

`SNAPSHOT_BIG_LITTLE_GUIDE.md` 已经明确指出：

```text
不要把 SD 卡写入、H.265 解码、JPEG 编码放在 DATAFIFO/RTP 线程里，
否则会增加卡顿风险。
```

即使大核在约 67ms 内收到 `READ_DONE`，小核也可能在复制后把数据放入另一个无界发送队列；这种积压不会体现在大核 `pending` 中。

## 7. AI 通道没有真正降低到 15fps

### 7.1 代码配置和运行结果不一致

`big_core/mpp_pipeline/mpp_pipeline.h` 中定义：

```c
#define VICAP_OUTPUT_FPS 15
```

`media_vicap.c` 又把该值写入：

```c
chn_attr.fps = VICAP_OUTPUT_FPS;
```

但日志中的 AI 线程实际约为 28.6fps，说明该配置没有让 AI dump 线程真正降到 15fps，或者该 SDK/工作模式下此字段没有产生预期的硬件降帧效果。

### 7.2 AI 线程成功取帧后没有主动限速

AI 线程当前流程为：

```text
kd_mpi_vicap_dump_frame
-> kd_mpi_sys_mmap
-> motion_detect_process
-> kd_mpi_sys_munmap
-> kd_mpi_vicap_dump_release
-> 立即获取下一帧
```

只有获取失败时才延时 10ms，成功路径没有任何检测帧率控制。因此只要 VICAP 能提供帧，AI 线程就会接近 30fps 运行。

### 7.3 当前内存访问存在优化空间

AI 帧来自 `VB_REMAP_MODE_NOCACHE` 内存，且每帧都执行：

- `calloc/free` 一个 handle。
- `mmap/munmap` Y 平面。
- 两次抽样访问。
- 一次完整的 307200 字节 `memcpy`。

这些操作会持续占用 CPU 和 DDR 带宽，是后续需要优化的问题。但是从本次日志的主 VENC 15fps 和 AI 约 28.6fps 看，它还不能解释数秒级积压。

## 8. `cur_packs` 仍需在大 IDR 场景中检查

`big_core/mpp_pipeline/media_venc.c` 中使用：

```c
output.pack_cnt = status.cur_packs;
if (output.pack_cnt == 0)
    output.pack_cnt = 1;
if (output.pack_cnt > VENC_MAX_PACKS)
    output.pack_cnt = VENC_MAX_PACKS;
```

而：

```c
#define VENC_MAX_PACKS MPP_MAX_STREAM_PACKS
#define MPP_MAX_STREAM_PACKS 8
```

当前代码会把超过 8 的 `status.cur_packs` 直接截断为 8。大画面突变产生的大 IDR 更可能暴露多 pack 情况。

本次日志没有出现正常运行期 `kd_mpi_venc_get_stream` 失败，因此还不能认定这里已经出错，但下一轮必须记录：

- `status.cur_packs` 当前值和最大值。
- IDR 帧的 `output.pack_cnt`。
- 每个 pack 的长度。
- 单个 stream 的总长度。
- `cur_packs > 8` 的发生次数。

不要仅凭本次日志把 `cur_packs` 判定为根因，也不能在没有数据的情况下排除它。

## 9. 当前目录与板端运行 ELF 仍然存在不一致

重新检查指定目录后，仍有两项可验证的不一致。

### 9.1 执行文件名称不同

板端运行的是：

```text
big_app_new.elf
```

当前目录中提供的是：

```text
big_core/big_app.elf
```

### 9.2 `IDR enable OK` 日志没有对应源码

板端日志包含：

```text
[MPP] VENC chn=0 IDR enable OK
```

但当前 `media_venc.c` 的创建和启动函数中没有该日志，也没有对应的 IDR enable 配置代码；目录中的 `big_app.elf` 同样没有该字符串。

### 9.3 当前诊断日志没有输出 `[motion:diag]`

当前 `motion_detect.c` 和目录中的 `big_app.elf` 都包含：

```text
[MPP] [motion:diag] ...
```

按源码逻辑，正常处理运动帧时应该约每秒打印一次，但上传的终端日志中一条都没有。

这表明板上运行的 `big_app_new.elf` 仍不是当前目录 `big_app.elf` 的完全对应版本，或者构建所使用的源码还有未同步的修改。

在继续修改前，应把板上实际运行的 `big_app_new.elf` 及其对应源码同步回工作目录，并记录 SHA256，避免继续比较不同版本。

## 10. 当前目录缺少端到端小核代码

当前 `mpp_pipeline--/mpp_pipeline--` 目录只包含大核 `big_core` 代码，没有实际运行的小核以下实现：

- DATAFIFO reader。
- NALU 物理地址映射和复制。
- RTP/H.265 分片发送。
- RTSP server。
- 快照队列。
- SD 卡写入。
- `ffmpeg` 转 JPG。
- 播放器前的发送队列和丢帧策略。

因此当前可以证明“大核没有形成 4 秒队列”，但还不能仅用这个目录确定 4 秒延迟在小核哪个函数中产生。

## 11. 推荐的最小 A/B 定位顺序

### 11.1 第一组：只关闭快照请求

保持以下内容不变：

- 使用新的高灵敏度运动检测算法。
- 保持运动检测线程开启。
- 保持 OSD 开启。
- 使用相同摄像头、网络、客户端和动态场景。

只临时设置：

```c
event->request_snapshot = 0;
```

或者只跳过：

```c
stream_export_request_snapshot(&snapshot_req);
```

结果判断：

- 延迟恢复到约 200ms：根因位于快照、强制 IDR 或小核快照处理链。
- 延迟仍然达到约 4 秒：继续进行 AI 帧率和检测耗时测试。

这是当前信息量最大、改动最小的一组测试。

### 11.2 第二组：保留检测但显式限制 AI 处理帧率

如果关闭快照后延迟仍高，再把运动检测显式限制到 10～15fps，而不是只依赖 `chn_attr.fps`。

需要同时记录：

- `ai_frame_try_get()` 平均/最大耗时。
- `motion_detect_process()` 平均/最大耗时。
- `ai_frame_release()` 平均/最大耗时。
- 实际 AI 处理 FPS。
- 主 VENC FPS。
- 端到端延迟。

如果显式降频后延迟恢复，说明主要问题是 AI dump、非缓存内存访问或共享 DDR 带宽，而不是检测结果触发链。

### 11.3 第三组：保留快照但取消立即强制 IDR

当前 `VENC_GOP=DST_FPS=15`，自然 IDR 间隔约一秒。

可以让快照请求等待下一个自然 IDR，而不对每个事件调用 `kd_mpi_venc_request_idr()`。

结果判断：

- 延迟恢复：强制 IDR和瞬时码流是主要触发点。
- 延迟仍高：检查小核快照复制、文件写入和 `ffmpeg` 处理。

## 12. 建议的解决方向

### 12.1 快照和主视频链路解耦

- 快照队列最多保留 1 个待处理任务。
- 快照线程忙时直接丢弃新快照。
- 同一时间最多运行一个 `ffmpeg`。
- 禁止在 DATAFIFO/RTP/RTSP 线程里写文件或转换图片。
- 快照线程使用更低优先级，必要时限制 CPU 占用。
- RTSP 视频实时性优先于快照完整率。

### 12.2 减少强制 IDR

- 优先等待约一秒内的自然 IDR。
- 对强制 IDR增加独立冷却时间，例如 3～5 秒。
- 连续运动期间只产生一次快照，不因短暂清除后再次马上触发。
- 记录 IDR 帧总字节数，确认网络能承受峰值。

### 12.3 显式限制 AI 处理帧率

- 不再假设 `VICAP_OUTPUT_FPS=15` 已经对 dump 通道生效。
- 使用单调时钟或帧序号控制运动检测实际执行到 10～15fps。
- 跳过检测的帧应尽快 release，避免占用 VICAP buffer。

### 12.4 减少运动检测的内存访问

当前比较时只使用每隔 4 像素的采样点，却保存完整的 `640x480` 参考帧。

后续可以只保存 `160x120=19200` 个采样亮度值，将参考缓冲从约 300KB 降到约 19KB，同时避免每帧完整复制 Y 平面。

该优化不会改变当前灵敏度计算公式，但能明显降低非缓存内存读取和内存复制开销。

### 12.5 实时流队列采用“保最新帧”策略

小核 RTSP/RTP 发送发生拥塞时：

- 不能无界保存旧帧。
- 应丢弃过期 P 帧。
- 必要时等待下一个 IDR 重新同步。
- 持续记录最老待发送帧的年龄。

实时流的目标是播放最新画面，而不是把所有历史帧延迟数秒后完整播放。

## 14. 结论优先级

按照当前代码和日志证据，排查优先级应为：

1. 运动事件触发的快照、立即强制 IDR和小核快照处理链。
2. 小核 RTSP/RTP 发送队列及播放器缓存。
3. AI 通道实际仍以约 30fps 处理，以及非缓存帧访问开销。
4. 大 IDR场景下 `status.cur_packs` 是否超过 8。
5. 运动检测函数本身的固定计算耗时。

当前最推荐的下一步不是继续修改灵敏度算法，而是保持新算法不变，只关闭 `request_snapshot` 做一轮动态场景 A/B 测试。该测试可以最快区分“检测计算问题”和“检测事件副作用问题”。
