# 视频流媒体服务器项目 - 团队协作指南

## 总体流水线怎么走？（大白话解释）

在 K230 上跑视频，绝对不能像 STM32 那样用 CPU 去一个字节一个字节地搬运像素（DCMI -> 数组 -> CPU 处理 -> 串口发出）。1080P 的数据量会瞬间把 CPU 撑死。我们需要建立一条**全自动的纯硬件流水线**，就像配置 STM32 的"定时器硬件触发 ADC，再触发 DMA 搬运"一样，全程不需要 CPU 瞎掺和。

**系统数据流转图：**

```text
               (硬件零拷贝主干道)
[摄像头] ---> [VI (输入模块)] ======(硬件 Bind 绑定)======> [VENC (H.265编码器)] ---> [NALU 压缩码流]
                  ||                                              ||                  |
                  ||(分流出一个低分辨率通道)                      ||(硬件自动叠加)    | (通过网络发走)
                  ||                                              ||                  v
           [VI_CH1 (640x480)]                             [OSD 硬件区域(Region)]    [LwIP Socket]
                  |                                               ^                   ^
                  v                                               | (指令触发)        |
           [进程B：AI 算法] --------(发现移动)----------------------+                   |
                                                                                      |
           [SD 卡写盘线程] <--------(保存快照)------------------------------------------+
```

- **队长**：负责搭建上面那条 `VI ===> VENC` 的"高速公路"（配置物理内存池，打通硬件通路）。
- **队员A**：站在高速公路出口，把 `NALU 压缩码流` 装进网络数据包扔出去，同时负责把快照按顺序慢悠悠地写进 SD 卡。
- **队员B**：站在旁路的低速道上，盯着 `VI_CH1` 的图像看，发现有东西动了，立刻按下一个"硬件开关"，让编码器在画面上盖上水印。

## 成员任务分配与配合

### 🧑‍💻 队长（攻坚 OS 与多媒体核心管线）

**你在做什么：**
你要初始化 K230 的 MPP（多媒体处理平台）系统，划分出一大块连续物理内存（VB 内存池）。然后配置摄像头的参数（分辨率、帧率），并通过一条叫 `kd_mpi_sys_bind` 的指令，把摄像头直接"焊"死在视频编码器上。随后，还要建立多进程之间的通信通道。

**这有什么用：**
这是整个系统的地基。你不把这条不需要 CPU 参与的硬件管线搭好，后续队员 A 和 B 连一帧 1080P 的数据都处理不动。

**核心黑话翻译：**

- **MMU与进程隔离**：在以前的 FreeRTOS 里，大家共用一块内存，传个全局变量指针就能通信。现在开启了 MMU，每个"进程"就像是一颗完全独立的 STM32 芯片。进程 A 的 `0x8000` 和进程 B 的 `0x8000` 映射的物理内存根本不是同一个地方，直接传指针会导致系统崩溃（段错误）。
- **MPP VB (Video Buffer) 内存池**：这就相当于你在 STM32 启动前，用 `#pragma` 强制保留了一块极其巨大的 DMA 专用循环 SRAM。所有外设都在这里面存取数据。
- **硬件 Bind**：这就相当于 STM32 里的 "定时器主从模式触发 ADC"。配置好之后，外设之间自动传数据，CPU 根本不需要写中断服务函数去搬运。
- **IPC (进程间通信) 消息队列**：既然不能用全局变量，这就是跨进程传数据的唯一合法途径。它相当于你在两颗不同的 STM32 之间接了一根 SPI 线，只能老老实实发打包好的结构体数据。

**怎么和队友配合：**
你需要把编码器压好的 H.265 NALU 码流的内存地址交接给 队员A。你需要利用 IPC 消息队列，把旁路低分辨率图像的"物理首地址"发送给 队员B。

### 🧑‍💻 队员 A（外设与网络流媒体）

**你在做什么：**
你需要写 UDP/TCP Socket 程序，把队长递给你的视频压缩包，加上网络协议头，通过网口或 Wi-Fi 发送到电脑上。同时，你需要挂载 FatFS，写一个优先级极低的后台线程，用来把图片写入 SD 卡。

**这有什么用：**
你是系统的"对外输出口"和"减震器"。如果没有你的低优先级写盘线程，SD 卡写入时的几百毫秒卡顿，会直接导致整个视频流断开、系统死机。

**核心黑话翻译：**

- **LwIP Socket 网络编程**：相当于你以前写 ESP8266 的 AT 指令发送函数，只不过现在换成了标准的 C 语言函数（`socket()`, `sendto()`），不需要你自己解析繁杂的底层网络电平信号。
- **异步写 SD 卡 (防卡死)**：回想一下，在 STM32 的中断服务函数（ISR）里绝对不能调用 `printf` 或 `HAL_Delay` 对吧？你会怎么做？你会把数据丢进 FreeRTOS 的消息队列，让主循环去慢慢打。这里的原理一模一样！绝对不能在发现移动的瞬间去写 SD 卡，必须把图片指针丢进 Mailbox，让一个专门的慢速线程去写（FatFS `f_write`）。

**怎么和队友配合：**
等待队长给你提供一个回调函数或者队列，里面会源源不断地吐出 `NALU` 缓冲区的指针和长度。你拿到后，直接调网络接口发走，绝对禁止使用 `memcpy` 把数据复制到你自己的数组里！

### 🧑‍💻 队员 B（算法与上层逻辑）

**你在做什么：**
你需要拿到小尺寸（如 640x480）的黑白或彩色图像数据，写一个简单的纯 C 语言算法对比前后两张图的差异（帧差法）。一旦检测到移动，立刻调用 K230 硬件的 OSD 接口，同时发信号给队员 A 让他存图。

**这有什么用：**
这是赛题的核心得分点（AI 移动侦测触发）。你的算法必须足够轻量，且触发 OSD 的动作必须在 1 毫秒内完成。

**核心黑话翻译：**

- **【严重纠错】OSD 字符叠加其实就是在显存数组里修改像素？**
  错！绝对大错特错！ 用 C 语言的 `for` 循环去 1080P 的数组里画字，CPU 缓存会当场爆炸，耗时会超过 50ms。
  这里的 OSD 相当于 STM32F429 里面的 LTDC (LCD-TFT 控制器) 硬件图层叠加 (DMA2D)。队长会提前在物理内存里放好一张写着 "Motion Detected" 的 PNG 透明图片。你要做的是修改一个硬件寄存器（Alpha 透明度）。设为 0 就是看不见，设为 255 硬件就会自动在下一帧盖上水印，CPU 耗时为 0 毫秒！
- **帧差法**：这就相当于你定时采集 ADC 的电压值，如果 `| 当前电压 - 上次电压 | > 阈值`，就判断按键按下了。放到图像上，就是 `| 当前帧的像素数组 - 上一帧的像素数组 | > 阈值`，统计有多少个像素发生了剧变。

**怎么和队友配合：**
从队长给你的 IPC 消息里提取出当前帧的物理地址。用完后，必须通知队长释放该帧内存。当你算出画面变动时，调用队长封装好的 `set_osd_alpha(255)` 函数。

## 第一周"不翻车"作战目标

> **纪律铁律：第一周内，禁止任何人尝试写"多进程"代码，全部在一个进程（单 main 函数）里多线程调通，先验证硬件逻辑！**

### 队长（目标：出码流）

- 参考《K230 MPP API 参考手册》，写出初始化 `VB 内存池` 的代码。
- 配置好 `VI_CH0` (1080P) 并 `Bind` 到 `VENC`。
- **验收标准**：程序跑起来后，串口终端能以 15fps 的速度，狂刷打印 `"Get NALU, Size: xxxx bytes"` 的日志，且运行 10 分钟不死机。

### 队员 A（目标：通网络）

- 学习 RT-Smart 的 socket 编程。先不要接队长的真实数据。自己在内存里伪造一段假数据（比如几百个 `0xAA`），写一个 UDP Sender。
- **验收标准**：能够在电脑端用网络抓包工具（Wireshark）或者网络调试助手，稳定接收到开发板发来的 UDP 数据包。

### 队员 B（目标：出算法与打水印）

- 在自己的电脑上用 Dev-C++ 或 Visual Studio，写一个纯 C 语言的函数 `int check_motion(uint8_t* img1, uint8_t* img2)`，验证你的帧差逻辑是能算出结果的。
- 在 K230 开发板上，学习调用 `kd_mpi_venc_set_region` API。
- **验收标准**：能成功在 K230 编码出的画面上（哪怕是黑屏死机画面），强制通过寄存器显示出一个死板的 "Motion Detected" 块。
