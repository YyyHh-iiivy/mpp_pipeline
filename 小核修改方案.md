# Wi‑Fi 省电、驱动和 AP 排队的详细解释与解决方案

## 1. 先明确：为什么 `sendto()` 很快，画面仍会越来越慢

你的小核日志中每帧 `send=1–2ms`，这只能说明 RTP 包很快从应用程序交给了 UDP 栈，不代表数据已经通过无线链路到达 PC。

实际路径是：

```text
VENC
  ↓
DATAFIFO
  ↓
小核 RTP 分片
  ↓
sendto()
  ↓
UDP socket 发送队列
  ↓
系统 qdisc/网络协议栈
  ↓
Wi‑Fi 驱动软件队列
  ↓
Wi‑Fi 固件聚合与重传队列
  ↓
AP 无线发送队列
  ↓
PC 网卡接收队列
  ↓
ffplay
```

Linux Wireless 文档明确指出，Wi‑Fi 设备通常包含软件队列、驱动/固件聚合队列和硬件 FIFO；802.11 聚合还可能等待多个帧后再发送。因此，用户态看到 `sendto()` 返回，并不能证明包已经离开无线芯片。[Wireless device queues](https://wireless.docs.kernel.org/en/latest/en/developers/bufferbloat.html)

你的代码中还设置了：

```c
#define RTP_SOCKET_SNDBUF    (128 * 1024)
```

位置在 [`rtp.h`](<F:/RT-Thead Smart Al/MY_Code/k230-lightweight-video-streaming-server-main-lowlatency-v2/small_core/lnc/rtp.h:34>)。

根据你之前日志：

- 平均视频码率约为 `2.04 Mbps`
- `600 ms` 的数据量约为 `153 KiB`
- `128 KiB` UDP 发送缓冲约等于 `0.5–0.6 秒` 的码流

所以你观察到的“250 ms 逐渐增长到约 600 ms 后稳定”，与发送队列、驱动队列和 AP 队列逐渐填满的现象非常吻合。

即使无线平均吞吐量只比视频平均输入速率低约 `0.5%–1%`，经过几十秒到几分钟也会积累出几百毫秒延迟。

## 2. Wi‑Fi 省电机制

### 2.1 省电如何增加延迟

Wi‑Fi 客户端进入省电模式后，不会一直保持接收状态。AP 会暂存发往客户端的数据，客户端等待 Beacon/DTIM 时间点醒来后再取数据。

Linux Wireless 文档说明，mac80211 动态省电可能在大约 `100ms` 没有流量后进入休眠；客户端通常需要等待 Beacon 或 DTIM 才能取回 AP 缓存的数据。[Dynamic power save](https://wireless.docs.kernel.org/en/latest/en/users/documentation/dynamic-power-save.html)

它可能产生：

- 50–100ms 左右的周期性抖动；
- 数据突然成批到达；
- RTP 包在 PC 端集中到达；
- 触发 ffplay 或 H.265 解码器等待完整帧。

不过需要注意：你的码流是 15fps，每约 66.7ms 就有一帧，因此“连续视频本身”可能让设备不满足 100ms 空闲条件。也就是说，省电值得测试，但它未必是唯一原因。

### 2.2 立即验证

在 K230 小核上执行：

```sh
command -v iw
iw dev wlan0 get power_save
iw dev wlan0 set power_save off
iw dev wlan0 get power_save
```

预期最后显示：

```text
Power save: off
```

`iw` 官方文档给出了查询和关闭 Wi‑Fi 电源管理的方法。[iw documentation](https://wireless.docs.kernel.org/en/latest/en/users/documentation/iw.html)

如果 `iw` 不存在：

- 当前系统可能没有安装 `iw`；
- 驱动可能不支持 nl80211；
- RT-Smart 的网络栈可能没有提供该接口。

这种情况下，不能把 `iwconfig wlan0 power off` 当成一定有效的替代方案，需要查询当前 K230 Wi‑Fi 驱动提供的私有命令或配置接口。

关闭省电后必须重新运行 10 分钟测试。不要只观察刚启动的前 30 秒。

### 2.3 省电相关解决方案

建议按以下顺序处理：

1. 关闭 K230 客户端省电。
2. 在 AP 端关闭该客户端的 WMM Power Save/U-APSD（如果 AP 提供该选项）。
3. AP 的 DTIM 暂时设置为 `1`，降低客户端等待周期。
4. 保持视频运行时不要频繁断开和重新关联 Wi‑Fi。
5. 关闭 K230 端可能存在的自动休眠或无线驱动节能策略。

省电关闭后，如果延迟仍然从 250ms 增长到 600ms，说明主要问题不是省电，而是队列积压或无线吞吐不足。

## 3. Wi‑Fi 驱动和固件队列

### 3.1 为什么驱动队列会积压

你当前 RTP 发送代码有两个特征：

```c
#define RTP_MAX_PAYLOAD      1200
#define RTP_PACKET_PACE_US   0
```

位置在 [`rtp.h`](<F:/RT-Thead Smart Al/MY_Code/k230-lightweight-video-streaming-server-main-lowlatency-v2/small_core/lnc/rtp.h:32>) 和 [`rtp.h`](<F:/RT-Thead Smart Al/MY_Code/k230-lightweight-video-streaming-server-main-lowlatency-v2/small_core/lnc/rtp.h:37>)。

也就是说，一帧 H.265 被分成很多 RTP 包后，基本会在几毫秒内全部提交给 socket。驱动和固件随后需要处理：

- Wi‑Fi MAC 层竞争；
- 空口重传；
- AMPDU 聚合；
- 速率自适应；
- 省电唤醒；
- AP 的 ACK 和转发。

如果应用程序每 66.7ms 突发一次，而无线实际需要 70–75ms 才能发送完成，那么每帧都会多积压几毫秒。延迟就会逐渐增加。

### 3.2 检查驱动状态

如果 K230 系统提供这些命令，可以运行：

```sh
iw dev wlan0 station dump
cat /proc/net/wireless
ip -s link show dev wlan0
tc -s qdisc show dev wlan0
```

重点查看：

- `signal`：信号强度；
- `tx bitrate`：当前实际发送速率；
- `tx retries`：重传次数；
- `tx failed`：发送失败次数；
- `backlog`：qdisc 排队长度；
- `dropped`：qdisc 丢包数；
- `overlimits`：限速或队列限制次数；
- `rx/tx errors`：网卡错误。

如果 K230 没有 `iw` 或 `tc`，可在 AP 或 Linux 网关上检查相同指标。

### 3.3 解决方案：不要把整个帧瞬间塞进 Wi‑Fi

最重要的代码级解决方案是：

#### 方案 A：先减小 socket 发送缓冲

当前：

```c
#define RTP_SOCKET_SNDBUF    (128 * 1024)
```

建议先试：

```c
#define RTP_SOCKET_SNDBUF    (32 * 1024)
```

如果延迟仍然会积累，再试：

```c
#define RTP_SOCKET_SNDBUF    (16 * 1024)
```

同时增加实际值检查：

```c
int requested = 32 * 1024;
int actual = 0;
socklen_t actual_len = sizeof(actual);

setsockopt(sock,
           SOL_SOCKET,
           SO_SNDBUF,
           &requested,
           sizeof(requested));

getsockopt(sock,
           SOL_SOCKET,
           SO_SNDBUF,
           &actual,
           &actual_len);

printf("[rtp] requested sndbuf=%d actual sndbuf=%d\n",
       requested,
       actual);
```

Linux 可能会对 `SO_SNDBUF` 做倍增，因此必须打印 `getsockopt()` 返回的实际值。

这里不是为了让发送更容易成功，而是为了让拥塞更早暴露为 `EAGAIN/EWOULDBLOCK`，然后丢弃当前旧帧，而不是把旧帧留在系统队列中。

#### 方案 B：增加 RTP pacing

现在：

```c
#define RTP_PACKET_PACE_US   0
```

可以先做快速 A/B 测试：

```c
#define RTP_PACKET_PACE_US   2000
```

如果仍然有突发，再试：

```c
#define RTP_PACKET_PACE_US   3000
```

但固定每包睡眠只是临时方案。更正确的做法是按目标码率计算间隔。

例如目标发送速率为 `2.5 Mbps`，一个 RTP payload 为 `1200 bytes`：

```text
interval =
    1200 * 8 * 1,000,000 / 2,500,000
    ≈ 3840 us
```

因此每个包大约间隔 3.8ms。

建议实现：

- 目标码率从实际统计码率的 1.1 倍开始；
- 允许最多 1–2 个包的短 burst；
- 超过 socket 队列阈值时丢弃当前帧；
- 不要为了发送旧帧而阻塞下一帧。

以你日志中的约 `2.04 Mbps` 为基础，可以先用 `2.5–3.0 Mbps` 做 pacing 测试。

#### 方案 C：监控 socket 当前排队字节数

如果 K230 网络栈支持 `SIOCOUTQ`，可以检查：

```c
int outq = 0;

if (ioctl(sock, SIOCOUTQ, &outq) == 0) {
    printf("[rtp] socket outq=%d bytes\n", outq);
}
```

如果 `outq` 随时间增加并在几万到十几万字节附近稳定，说明问题就是 socket/驱动侧排队。

如果该 ioctl 在 RT-Smart 中不可用，就只能通过：

- `EAGAIN/EWOULDBLOCK` 次数；
- Wireshark 的 RTP 到达时间；
- 应用层发送时间戳；
- AP 的 station 统计；

间接判断。

## 4. AP 排队和 Bufferbloat

### 4.1 AP 为什么会让局域网视频延迟增加

AP 不只是“转发无线包”，还要负责：

- 多个客户端竞争空口；
- 低速客户端占用更多 airtime；
- Wi‑Fi 重传；
- AMPDU 聚合；
- WMM 优先级队列；
- 无线到有线方向的转发；
- 有线接口和交换芯片缓冲。

如果 AP 允许队列持续变长，就会出现 Bufferbloat：

- 刚开始队列为空，延迟约 250ms；
- 一段时间后队列变长，延迟升到 600ms；
- 队列满后，延迟保持在一个较高水平；
- 由于 UDP 没有重传，丢包和延迟可能同时出现。

### 4.2 AP 侧检查

如果 AP 是 OpenWrt/Linux，可以查看：

```sh
iw dev wlan0 station dump
tc -s qdisc show
```

如果 AP 是普通家用路由器，则重点做以下物理对照：

1. K230 连接 5GHz，而不是 2.4GHz。
2. K230 靠近 AP。
3. 固定一个干扰较小的信道。
4. 暂时断开其他低信号、低速率客户端。
5. 使用独立 SSID 做测试。
6. 让 PC 接 AP 的有线口，K230 使用 Wi‑Fi。
7. 再把 K230 换成有线连接，比较结果。

如果：

- K230 Wi‑Fi → PC Wi‑Fi：延迟累增；
- K230 Wi‑Fi → PC 有线：仍累增；
- K230 有线 → PC 有线：稳定；

则基本可以确定问题在 K230 到 AP 的无线链路或 AP 无线队列。

### 4.3 AP 上使用 SQM/FQ-CoDel

如果 AP 的瓶颈是 WAN、AP 到交换机链路或路由转发队列，可以使用 SQM。FQ-CoDel 的作用是限制队列长度并主动丢弃过期数据，而不是无限缓存。[tc-fq_codel(8)](https://man7.org/linux/man-pages/man8/tc-fq_codel.8.html)

临时测试：

```sh
tc qdisc replace dev wlan0 root fq_codel \
    limit 256 \
    target 5ms \
    interval 100ms
```

检查统计：

```sh
tc -s qdisc show dev wlan0
```

如果 AP 使用 OpenWrt，可优先使用 CAKE/SQM；OpenWrt 官方文档建议 SQM 用于控制 Bufferbloat，并说明使用 SQM 时需要注意硬件流量卸载可能绕过队列管理。[OpenWrt SQM](https://openwrt.org/docs/guide-user/network/traffic-shaping/sqm)

注意：

- 如果 K230 和 PC 在同一个普通 AP 的局域网内，SQM 不一定能直接控制无线芯片内部队列；
- 不要把 WAN SQM 当成 Wi‑Fi 无线队列的万能解决方案；
- 先确认实际瓶颈在哪个接口；
- 不建议一开始就关闭 WMM，WMM 本身是 Wi‑Fi QoS 的基础。

## 5. 当前代码中的 QoS 设置也可能没有真正生效

你当前设置：

```c
#define RTP_IP_TOS_LOWDELAY  0x10
```

并在 [`rtp.c`](<F:/RT-Thead Smart Al/MY_Code/k230-lightweight-video-streaming-server-main-lowlatency-v2/small_core/src/rtp.c:41>) 中使用 `IP_TOS`。

`0x10` 是旧式 IP TOS 的 low-delay 标志，并不保证 AP 会把它映射为 WMM 视频队列。很多现代网络设备只根据 DSCP 做 QoS，或者直接忽略该字段。

可以做一个可选实验：

```c
int tos = 34 << 2;   /* DSCP AF41 = 0x88 */
setsockopt(sock, IPPROTO_IP, IP_TOS, &tos, sizeof(tos));
```

然后在 AP 或 Wireshark 中检查 DSCP 是否保留。

这不是第一优先级，因为错误的 QoS 映射可能让视频进入更高优先级队列，反而挤压其他业务。应先完成普通 UDP、5GHz、有线和 pacing 对照，再测试 DSCP。

## 6. 你还需要处理丢包后的恢复延迟

当前大核配置中：

```c
#define VENC_FORCE_IDR_ENABLE 0
```

这意味着小核发送失败后发出的 IDR 请求可能被跳过，只能等待下一个 GOP。

当前 GOP 为 8，15fps 下等待下一个随机接入帧最长约：

```text
8 / 15 ≈ 533ms
```

因此一次 RTP 分片丢失可能带来：

- 当前 H.265 帧无法完整解码；
- 小核进入等待随机接入帧；
- 需要等待下一个 IDR；
- 延迟瞬间增加约 0.5 秒。

这不是“逐渐累增”的唯一原因，但会让累增后的延迟更难恢复。

建议：

1. 确认 `kd_mpi_venc_request_idr()` 是否可用。
2. 如果可用，打开 `VENC_FORCE_IDR_ENABLE`。
3. 如果不可用，考虑使用较短 GOP，但要评估码率增加。
4. RTP 丢包后清空旧帧，只保留下一帧完整 IDR。
5. 不要等待旧 RTP 包全部发送完再恢复。

## 7. 推荐的实际排查顺序

### 第一步：关闭省电

```sh
iw dev wlan0 set power_save off
```

保持其他参数不变，运行 10 分钟。

### 第二步：做有线对照

如果条件允许：

```text
K230 有线 → PC 有线
```

如果有线稳定，而 Wi‑Fi 累增，问题就在无线链路/AP，不必继续优先怀疑 ffplay。

### 第三步：减小发送缓冲

从：

```c
RTP_SOCKET_SNDBUF = 128 * 1024
```

改为：

```c
RTP_SOCKET_SNDBUF = 32 * 1024
```

记录：

- `EAGAIN` 次数；
- RTP drop 次数；
- 每帧 `send_cost_ms`；
- 延迟是否从 600ms 回到 250–350ms。

### 第四步：加入 pacing

先用：

```c
RTP_PACKET_PACE_US = 2000
```

如果延迟继续增加，再采用按码率计算的 token-bucket pacing，而不是固定睡眠。

### 第五步：检查无线质量

重点看：

```sh
iw dev wlan0 station dump
```

如果 `tx retries`、`tx failed` 很高，优先处理：

- 5GHz；
- 信道；
- 天线；
- AP 距离；
- 其他低速客户端；
- AP 固件。

### 第六步：处理跨核旧帧

当前小核 `age=0ms` 不可信。小核不应直接比较大核和小核各自的 `CLOCK_MONOTONIC`。

应改成：

- 以 `frame_pts` 为媒体时间；
- 小核记录收到第一帧时的本地时间；
- 用 PTS 差值估算媒体应到达时间；
- 超过 150–220ms 的旧帧直接丢弃；
- 保留最新完整 IDR 和 VPS/SPS/PPS。

## 推荐初始参数

建议先用下面这一组做实验：

```c
#define RTP_SOCKET_SNDBUF    (32 * 1024)
#define RTP_PACKET_PACE_US   2000
#define RTP_MAX_PAYLOAD      1200
```

并同时：

```sh
iw dev wlan0 set power_save off
```

如果延迟仍然累增，再将 pacing 改为按 `2.5–3.0 Mbps` 目标码率计算，而不是继续增大 socket 缓冲。

## 验收标准

每种配置至少运行 10 分钟，并记录：

- 初始延迟；
- 1分钟延迟；
- 5分钟延迟；
- 10分钟延迟；
- Wi‑Fi 信号；
- tx retries；
- RTP 丢包；
- RTP drop；
- socket 队列；
- 大核 `pending/datafifo_full`。

判断标准：

- 关闭省电有效：说明存在客户端休眠/DTIM 延迟；
- 5GHz或有线有效：说明是无线 airtime/AP/驱动问题；
- 减小 `SO_SNDBUF` 有效：说明原先 socket 队列过深；
- pacing 有效：说明是 RTP 分片突发造成的无线队列膨胀；
- 只有修正 PTS/旧帧丢弃有效：说明是跨核时间戳和应用层积压；
- 所有无线测试都累增但有线稳定：不要继续调整 ffplay，应优先修改板端发送策略和 AP 配置。
