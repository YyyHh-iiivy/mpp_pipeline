# 快照联动修改说明

本文说明本次围绕“检测到移动后调用 OSD 开关接口或投递模拟 IPC 事件，并与队员 A 的快照队列接口打通”的修改内容。

## 修改目标

- AI 运动检测确认后继续触发 OSD 显示。
- 同一个运动事件同时登记一次快照请求。
- 大核不直接写 SD 卡、不复制图片、不做 JPEG 编码，只在后续 DATAFIFO 码流消息中设置快照标志。
- 小核 A 侧按 `SNAPSHOT_BIG_LITTLE_GUIDE.md` 约定，在 `reserved & MPP_NALU_IPC_FLAG_SNAPSHOT` 时复制该帧码流并投递快照队列。
- 保证主 VENC -> DATAFIFO -> 小核 READ_DONE -> 大核 release 的主链路不会因为快照请求卡死。

## 主要改动

### 1. 共享 IPC 协议

文件：`mpp_pipeline--/mpp_pipeline--/big_core/mpp_pipeline/mpp_nalu_ipc.h`

新增：

```c
#define MPP_NALU_IPC_FLAG_SNAPSHOT  (1U << 0)
```

该 bit 复用 `mpp_nalu_ipc_msg.reserved` 字段，和 `SNAPSHOT_BIG_LITTLE_GUIDE.md` 的约定一致，不改变 DATAFIFO item 大小和结构布局。

### 2. 运动事件置位快照请求

文件：`mpp_pipeline--/mpp_pipeline--/big_core/mpp_pipeline/motion_adapter.c`

运动事件确认后：

```c
event->request_snapshot = 1;
```

因此一次有效运动事件会同时携带：

- `event_id`
- `motion_score`
- `osd_duration_ms`
- `request_snapshot`

### 3. AI 线程串起 OSD 和快照请求

文件：`mpp_pipeline--/mpp_pipeline--/big_core/mpp_pipeline/motion_thread.c`

闭环顺序为：

```text
motion_adapter_process()
-> ai_frame_release()
-> osd_set_motion_visible()
-> stream_export_request_snapshot()
```

关键点：快照请求在 `ai_frame_release()` 之后才登记，不会因为 OSD 或快照逻辑占住 AI dump frame。

### 4. 码流导出层挂起快照请求

文件：`mpp_pipeline--/mpp_pipeline--/big_core/mpp_pipeline/stream_export.c`

新增 `stream_export_request_snapshot()`，内部维护一个最多 4 个元素的快照请求 FIFO。

DATAFIFO 模式下，后续提交 VENC stream 时：

```text
stream_export_submit_venc_stream()
-> stream_export_select_snapshot_flags()
-> nalu_ipc_submit_stream(..., snapshot_flags)
```

如果存在待快照请求，会优先等待带 `K_VENC_HEADER` 的码流包；超出等待帧数后兜底标记后续帧，避免请求一直悬挂。

本地日志模式下不会写 DATAFIFO，只打印模拟事件：

```text
Snapshot mock IPC event: event_id=... reserved=0x1 hint=...
```

### 5. DATAFIFO 消息写入 reserved 标志

文件：`mpp_pipeline--/mpp_pipeline--/big_core/mpp_pipeline/stream_nalu_ipc.c`

`nalu_ipc_build_msg()` 现在会执行：

```c
msg->reserved = flags;
```

成功投递带快照标志的消息后，大核打印：

```text
[big] snapshot flag set seq=... pts=... reserved=0x1
```

这就是队员 A 小核侧快照队列的触发点。

### 6. 主流程初始化顺序调整

文件：`mpp_pipeline--/mpp_pipeline--/big_core/mpp_pipeline/mpp_pipeline.c`

现在顺序为：

```text
vicap_start()
-> stream_export_init(STREAM_EXPORT_DATAFIFO)
-> ai_motion_thread_start()
-> rt_thread_create(stream_thread)
```

这样 AI 线程启动后如果立刻检测到运动，快照出口已经初始化完成。

## 主链路防卡死检查

主链路释放责任仍然保持原逻辑：

```text
stream_thread()
-> kd_mpi_venc_get_stream()
-> stream_export_submit_venc_stream()
-> 成功 DATAFIFO 接管：release_by_caller = K_FALSE
-> DATAFIFO 失败或本地日志模式：release_by_caller = K_TRUE
-> release_by_caller 为真时立即 kd_mpi_venc_release_stream()
```

因此：

- DATAFIFO 满时不会卡住 VENC buffer，当前帧会返回给调用方释放。
- 快照请求只影响 `reserved` 标志，不会在大核做 SD 卡写入。
- 大核不会把 DATAFIFO 物理地址直接交给快照线程。
- 小核必须在 `DATAFIFO_CMD_READ_DONE` 前复制被标记帧的数据，之后再让大核释放 VENC stream buffer。

## 快照保存路径

当前仓库只包含大核侧代码。实际保存路径仍在队员 A 小核侧：

```text
大核 AI 事件
-> OSD 显示
-> stream_export_request_snapshot()
-> 后续 VENC stream 的 msg.reserved |= MPP_NALU_IPC_FLAG_SNAPSHOT
-> 小核 DATAFIFO/RTP 线程读取 reserved
-> 小核在 READ_DONE 前复制编码数据
-> 小核快照队列
-> 快照线程写入 /sdcard/snapshots
```

大核侧不直接生成 `/sdcard/snapshots/*.jpg`。小核侧应按指南输出类似：

```text
[datafifo] seq=123 ... flags=0x1 ...
[snapshot] queued len=... pts=... reason=datafifo-reserved
[snapshot] saved /sdcard/snapshots/snapshot_000001.jpg ...
```

## 已做验证

已完成源码级检查：

- `stream_export_init()` 在 `ai_motion_thread_start()` 之前。
- `ai_frame_release()` 在 OSD 和快照请求之前。
- `msg->reserved = flags` 已写入 DATAFIFO 消息。
- `nalu_ipc_release_callback()` 仍负责小核 `READ_DONE` 后释放 VENC stream。
- 没有残留 `request_snapshot = 0`。
- 没有旧的 `nalu_ipc_submit_stream(chn, stream)` 调用。

本机限制：

- 当前 Windows 环境没有 `gcc`。
- 当前 Windows 环境没有 `scons`。
- WSL 未安装可用发行版。

因此尚未在本机完成主机侧编译或板端 SCons 构建；最终仍需要在 K230 SDK 环境中执行构建，并在板端联调小核快照线程。

## 板端建议验证

1. 编译大核 `big_app.elf`。
2. 启动小核 RTSP/DATAFIFO 程序。
3. 观察大核是否打印：

```text
Snapshot request queued: event_id=...
[big] snapshot flag set seq=... pts=... reserved=0x1
```

4. 观察小核是否打印：

```text
[datafifo] seq=... flags=0x1
[snapshot] queued ...
[snapshot] saved /sdcard/snapshots/snapshot_000001.jpg
```

5. 持续运行并确认视频流不断、VENC buffer 不耗尽、快照目录文件名递增且不覆盖。
