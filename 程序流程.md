# 程序流程

本文用于同步记录当前 `big_core/mpp_pipeline` 的代码结构、调用关系和数据流。每次新增或修改代码文件后都应同步更新。

文件命名前缀按任务目标划分：

- `media_*`：大核 MPP 底层媒体能力，包括 VB、VICAP、VENC、硬件绑定、OSD 和资源清理。
- `stream_*`：H.265/NALU 码流出口和小核 RTSP/RTP 对接准备。
- `motion_*`：AI 低清帧、运动检测算法契约和运动事件闭环。
- `mpp_pipeline.*`、`mpp_types.h`、`mpp_nalu_ipc.h` 保持为总入口和公共契约。
- `big_core/SConscript` 是当前应用实际生效的顶层源码白名单；新增的 `stream_freshness.c` 同时登记在顶层和模块内 `SConscript`，不使用 glob。

## 1. 函数抽象

### big_core/mpp_pipeline/mpp_types.h

- `AI_GRAY_MAX_WIDTH / AI_GRAY_MAX_HEIGHT -> compile_time_contract`
  输入：无运行期输入。
  输出：AI 灰度检测输入的最大宽高，当前为 `640x480`；`mpp_pipeline.h` 的 `AI_WIDTH`、`AI_HEIGHT` 与其保持一致。

- `ai_gray_frame_view(frame_id, timestamp_ms, width, height, stride, y) -> ai_luma_view`
  输入：AI 低清通道 dump 出来的 NV12 Y 平面元数据。
  输出：运动检测只读灰度视图；`y` 指向当前进程可读的 Y 平面，调用方不转移所有权，处理完仍由 `ai_frame_release()` 释放。`stride` 是相邻两行 Y 数据起点之间的字节跨度，可能因为 DMA 对齐大于 `width`，算法遍历第 `row` 行时必须使用 `y + row * stride`。

- `motion_detect_result(is_motion, motion_score) -> detection_result`
  输入：运动检测算法填充。
  输出：`is_motion != 0` 表示检测到运动；`motion_score` 为算法给出的强度分数。默认弱实现使用变化像素百分比，当前 B 侧强实现使用抽样变化点千分比，适配层只透传分数，不依赖具体量纲做阈值判断。

### big_core/mpp_pipeline/motion_detect.h

- `motion_detect_process(frame: const ai_gray_frame_view *, result: motion_detect_result *) -> k_s32`
  输入：AI 灰度帧和检测结果输出结构。
  输出：`0` 表示处理成功；当前由 `motion_detect.c` 提供 B 侧强符号实现，并覆盖 `motion_detect_default.c` 的弱符号示例。B 侧实现只负责检测并填写 `result`，不能直接操作 OSD、VICAP、VENC、RT-Thread 线程或释放帧资源。

### big_core/mpp_pipeline/mpp_pipeline.c

- `main(void) -> int`
  输入：无；读取编译期宏配置，如 `SENSOR_TYPE`、`ENC_WIDTH`、`ENC_HEIGHT`、`DST_FPS`、`VICAP_OUTPUT_FPS`、`OUTPUT_BUF_CNT=3`、`STREAM_EXPORT_DATAFIFO`。
  输出：程序退出码；`0` 表示初始化、运行、清理完成，非 `0` 表示失败。

- `sig_handler(signo: int) -> void`
  输入：`SIGINT` 或 `SIGTERM` 等信号编号。
  输出：无；只设置 `g_stop_signal` 和 `g_running = 0`，触发主线程和 AI 线程进入退出流程。码流线程由独立的 `g_stream_running` 控制，会继续 drain VENC，直到 cleanup 明确停止。该函数不打印日志，避免在信号上下文里调用 `printf` 导致 Ctrl+C 退出卡住。

### big_core/mpp_pipeline/media_vb.c

- `vb_init(void) -> k_s32`
  输入：无；使用 `FRAME_BUF_SIZE`、`STREAM_BUF_SIZE`、`AI_CHN_BUF_SIZE` 和各 pool 块数宏。
  输出：`0` 表示 VB pool 初始化成功；非 `0` 表示 SDK 调用失败。低延迟 RTSP 档使用 `INPUT_BUF_CNT=3`、`OUTPUT_BUF_CNT=3`；两个 DATAFIFO pending slot 满足实测约 `100ms` 的 READ_DONE 生命周期，额外一个 VENC 输出块留给当前 get/release 周期。

### big_core/mpp_pipeline/media_venc.c

- `venc_create_chn(chn: k_u32, bitrate: k_u32) -> k_s32`
  输入：VENC 通道号、目标码率 kbps。
  输出：`0` 表示 H.265 CBR 编码通道创建成功，并把 `g_status` 推进到 `STATUS_VENC_CREATED`；非 `0` 表示创建失败。该函数按 `SRC_FPS=30`、`DST_FPS=15` 配置 CBR 码控，默认 `VENC_GOP=8`，自然关键帧间隔约 `533ms`；不启用运行期主动 IDR，也不启动通道。创建成功后打印 FPS、GOP、VB 数量和 `70/35ms` 新鲜度阈值，供板端核对 ELF 配置。

- `venc_start_chn(chn: k_u32) -> k_s32`
  输入：VENC 通道号。
  输出：`0` 表示 VENC 编码通道启动成功，并把 `g_status` 推进到 `STATUS_VENC_STARTED`；非 `0` 表示失败。

- `stream_thread(arg: void *) -> void`
  输入：RT-Thread 线程参数，转换为 VENC 通道号。
  输出：无；在 `g_stream_running` 为真时循环获取 VENC stream。每轮在查询 VENC 前先调用 `stream_export_flush()`，使小核 `READ_DONE` 对应的 release callback 不依赖下一帧有效码流也能推进。`kd_mpi_venc_get_stream()` 成功但返回 `pack_cnt=0` 时不进入 freshness、统计或导出层，也不调用零包 release；线程限频输出 `VENC empty stream` 的 `cur_packs/release_pic_pts/end_of_stream/pending` 后等待 `5ms` 重试。有效 stream 取首个非 Header 且非零 PTS 的 pack 送入 `stream_freshness_observe()`；连续两帧源时钟漂移超过 `70ms` 后直接 release 迟到 stream，直到漂移降到 `35ms`。周期日志输出 `pending/source_drift_ms/max_drift_ms/stale_drop`，每轮正常释放后短暂 `rt_thread_mdelay(1)` 主动让出调度。

### big_core/mpp_pipeline/stream_freshness.h / stream_freshness.c

- `stream_freshness_init(tracker: stream_freshness_tracker *) -> void`
  输入：新鲜度状态对象。
  输出：无；清零基线、连续迟到计数、追帧状态、最大漂移和丢帧计数。

- `stream_freshness_observe(tracker, pts_us, now_ms, drift_ms, drop) -> k_s32`
  输入：非 Header 帧的微秒 PTS、当前单调时钟毫秒值和输出指针。
  输出：`0` 表示完成判断；PTS 为 `0` 时不建立基线。首个有效 PTS 建立 `(base_pts_us, base_now_ms)`，后续计算 `source_drift_ms=(now-base_now)-(pts-base_pts)/1000`。连续 `2` 帧超过 `70ms` 才进入追帧并令 `drop=K_TRUE`；追帧期间漂移降至 `35ms` 即恢复。PTS 或单调时钟回退时安全重建基线。

- `stream_thread_stop(void) -> void`
  输入：无；由 `pipeline_cleanup()` 在停止 VICAP/解绑/反初始化 VICAP 后调用。
  输出：无；设置 `g_stream_running = 0`，有限等待 `g_stream_exit_sem`，成功后删除 stream 退出信号量并清空 `g_stream_tid`。

### big_core/mpp_pipeline/media_vicap.c

- `vicap_try_config(sensor_type: k_vicap_sensor_type) -> k_s32`
  输入：传感器类型。
  输出：`0` 表示 sensor 查询、device 配置、主通道和 AI 通道配置、VICAP 初始化成功。主通道和 AI 通道都通过 `vicap_set_channel_attr()` 下发 `VICAP_OUTPUT_FPS=15`，避免继续按 sensor 原始 30fps 往后级送帧。

- `vicap_config(sensor_type: k_vicap_sensor_type) -> k_s32`
  输入：首选传感器类型。
  输出：`0` 表示首选或 fallback 配置成功；非 `0` 表示全部失败。

- `vicap_start(void) -> k_s32`
  输入：无。
  输出：`0` 表示 VICAP stream 启动成功。

### big_core/mpp_pipeline/media_bind.c

- `vi_bind_venc(void) -> k_s32`
  输入：无；使用固定 `VICAP_CHN` 和 `VENC_CHN`。
  输出：`0` 表示主通道 `VI -> VENC` 硬件绑定成功。

### big_core/mpp_pipeline/stream_export.c

- `stream_export_init(mode: stream_export_mode) -> k_s32`
  输入：导出模式，当前常用 `STREAM_EXPORT_DATAFIFO` 或 `STREAM_EXPORT_LOCAL_LOG`。
  输出：`0` 表示导出层初始化成功；DATAFIFO 模式会同时初始化 `nalu_ipc_module`。初始化时清空快照请求 FIFO，确保 AI 线程启动后看到的是新的导出状态。

- `stream_export_request_snapshot(request: const snapshot_request_msg *) -> k_s32`
  输入：运动事件生成的快照请求，包含 `event_id`、`frame_time_ms`、`source_chn` 和路径提示。
  输出：`0` 表示请求已进入最多 `4` 个元素的快照 FIFO；非 `0` 表示队列已满，本次请求被丢弃。该函数不请求主动 IDR、不复制码流、不写 SD 卡；入队请求由后续码流处理等待编码器按 GOP 自然产生的 `K_VENC_HEADER`。

- `stream_export_submit_venc_stream(chn: k_u32, stream: const k_venc_stream *, release_by_caller: k_bool *) -> k_s32`
  输入：VENC 通道号、`kd_mpi_venc_get_stream()` 得到的 stream、释放责任输出指针。
  输出：`0` 表示导出层处理成功；本地日志模式限频打印 NALU 并设置 `release_by_caller = K_TRUE`，DATAFIFO 成功接管时设置 `release_by_caller = K_FALSE`，DATAFIFO 投递失败时返回错误并要求调用方立即释放。若存在待快照请求，会优先等待带 `K_VENC_HEADER` 的 stream，超过 `VENC_GOP + 2` 帧后兜底给后续帧设置 `MPP_NALU_IPC_FLAG_SNAPSHOT`。

- `stream_export_flush(void) -> k_s32`
  输入：无。
  输出：DATAFIFO 模式下触发 `nalu_ipc_flush()` 处理小核 `READ_DONE` 回调；本地日志模式直接返回 `0`。

- `stream_export_get_pending_count(void) -> k_u32`
  输入：无。
  输出：DATAFIFO 模式下返回 `nalu_ipc_get_pending_count()`；本地日志模式或导出层未初始化时返回 `0`。该函数只暴露 pending 深度给统计日志，不让 `media_venc.c` 直接依赖 DATAFIFO 细节。

- `stream_export_deinit(void) -> void`
  输入：无。
  输出：无；DATAFIFO 模式下会反初始化 `nalu_ipc_module`，并清空尚未投递的快照请求。

### big_core/mpp_pipeline/stream_nalu_ipc.c

- `nalu_ipc_init(void) -> k_s32`
  输入：无；使用 DATAFIFO writer 参数 `entries=3`、`NALU_IPC_PENDING_MAX=2`、`item_size=512`。
  输出：`0` 表示 DATAFIFO writer 打开成功，并打印给小核使用的 `phy_addr` 与 `pending_max/fifo_entries/output_buf/drain_max_passes` 配置。初始化时清零 pending 表、高水位、提交序号、drop/full/write 统计、最新提交 seq gap 和最近一次 `READ_DONE` 耗时记录。

- `nalu_ipc_submit_stream(chn: k_u32, stream: const k_venc_stream *, flags: k_u32) -> k_s32`
  输入：VENC 通道号、`kd_mpi_venc_get_stream()` 得到的 stream，以及写入 `mpp_nalu_ipc_msg.reserved` 的标志位。
  输出：`0` 表示已把 NALU 描述符写入 DATAFIFO；非 `0` 表示未投递成功，导出层会要求 `media_venc.c` 立即 release 当前 stream。函数每次尝试都会先分配递增 `seq` 和 `submit_time_ms`，再通过 `nalu_ipc_drain_releases()` 有界处理小核 `READ_DONE` 回调。若 DATAFIFO 可写空间不足则丢当前帧；若两个 pending slot 已满，会输出 slot 年龄、再 drain 并重试一次，仍无空位才记录 `pending_full_after_drain`。每次 pending 增长同时更新 `pending_high_water`。

- `nalu_ipc_drain_releases(void) -> k_s32`
  输入：无。
  输出：DATAFIFO `write(NULL)` 的结果；最多执行 `NALU_IPC_DRAIN_MAX_PASSES=4` 次，用于主动拉取 reader 侧 `READ_DONE` 通知并触发 release callback。若某次调用后 `g_pending_count` 没有下降，则停止本轮 drain，避免在异常状态下空转。

- `nalu_ipc_log_pending_slots(reason: const char *, avail_write_len: k_u32) -> void`
  输入：诊断原因和当前 DATAFIFO 可写长度。
  输出：无；在 pending 满等异常路径打印每个 pending 凭据的 `seq`、`chn`、`age_ms`、`total_len` 和 `avail`，用于区分“小核没有 READ_DONE”和“大核没有及时处理 release callback”。

- `nalu_ipc_flush(void) -> k_s32`
  输入：无。
  输出：`nalu_ipc_drain_releases()` 的结果；用于处理小核 `READ_DONE` 后触发的释放回调。

- `nalu_ipc_release_callback(datafifo_item: void *) -> void`
  输入：DATAFIFO release callback 传回的 `mpp_nalu_ipc_msg` 描述符指针。
  输出：无；用 `seq + chn` 找回 pending 表中的延迟释放凭据，计算 `nalu_ipc_now_ms() - submit_time_ms` 得到 `read_done_age_ms`，再重建 `k_venc_stream` 调用 `kd_mpi_venc_release_stream()`。callback 会保存最近一次 `read_done_seq/read_done_age_ms`，并对首个、周期性或超过 `100ms` 的 `READ_DONE` 输出带 `seq` 的诊断日志。

- `nalu_ipc_get_pending_count(void) -> k_u32`
  输入：无。
  输出：当前已交给 DATAFIFO、但尚未收到小核 `READ_DONE` 的 stream 数量，目标稳定在 `0-2`，短时最多 `3`。

- `nalu_ipc_get_phy_addr(void) -> k_u64`
  输入：无。
  输出：DATAFIFO ring buffer 物理地址，供小核 `kd_datafifo_open_by_addr()` 使用。

- `nalu_ipc_deinit(void) -> void`
  输入：无。
  输出：无；flush DATAFIFO，强制释放未被小核 `READ_DONE` 的 pending stream，并关闭 DATAFIFO。

### big_core/mpp_pipeline/motion_adapter.c

- `motion_adapter_init(void) -> k_s32`
  输入：无。
  输出：`0` 表示运动适配层初始化完成，并重置事件序号、连续运动帧计数、连续静止帧计数、锁存状态和上次事件时间。

- `motion_adapter_process(frame: const ai_gray_frame_view *, event: motion_event_msg *, has_event: k_bool *) -> k_s32`
  输入：AI 灰度帧、事件输出、是否有事件输出标志。
  输出：`0` 表示处理成功；内部调用 `motion_detect_process()`，先对算法输出做消抖，再把确认后的运动转换成 `motion_event_msg`。消抖规则为连续 `3` 帧运动才触发、连续 `5` 帧静止才解除锁存、同一段持续运动只发送一次事件、两次事件至少间隔 `1000ms`。确认后的事件会设置 `request_snapshot = 1` 和 `osd_duration_ms = 1000`，让 AI 线程同时触发 OSD 和快照申请；该函数不取帧、不释放帧、不直接控制 OSD。

- `motion_adapter_deinit(void) -> void`
  输入：无。
  输出：无；输出运动适配层反初始化日志。

### big_core/mpp_pipeline/motion_detect.c

- `motion_detect_validate_frame(frame: const ai_gray_frame_view *) -> k_s32`
  输入：AI 灰度帧视图。
  输出：`0` 表示 `frame`、`y`、宽高、`stride`、最大尺寸和 `width * height` 乘法范围都满足 B 侧算法要求；非 `0` 表示非法输入。该函数为文件内静态辅助函数。

- `motion_detect_store_frame(frame: const ai_gray_frame_view *) -> void`
  输入：AI 灰度帧视图。
  输出：无；按 `stride` 逐行复制有效 `width` 字节到上一帧缓存，缓存按 `width * height` 紧密排列，不保存每行 padding。该函数为文件内静态辅助函数。

- `motion_detect_process(frame: const ai_gray_frame_view *, result: motion_detect_result *) -> k_s32`
  输入：AI 低清通道 NV12 的 Y 平面视图。
  输出：`0` 表示处理成功；当前生产构建使用该强符号实现覆盖默认弱符号。算法首帧或分辨率变化时只重建上一帧缓存，后续每隔 `4` 个像素采样一次，统计亮度差大于等于 `25` 的采样点比例，并用千分比写入 `motion_score`；达到 `15` 时置 `is_motion = 1`。

### big_core/mpp_pipeline/motion_detect_default.c

- `motion_validate_frame(frame: const ai_gray_frame_view *) -> k_s32`
  输入：AI 灰度帧视图。
  输出：`0` 表示 `frame`、`y`、宽高、`stride` 和最大尺寸都满足默认算法要求；非 `0` 表示非法输入。该函数为文件内静态辅助函数。

- `motion_copy_baseline(frame: const ai_gray_frame_view *) -> void`
  输入：AI 灰度帧视图。
  输出：无；按 `stride` 逐行复制有效 `width` 字节到内部上一帧基线，不复制每行 padding。该函数为文件内静态辅助函数。

- `motion_detect_process(frame: const ai_gray_frame_view *, result: motion_detect_result *) -> k_s32`
  输入：AI 低清通道 NV12 的 Y 平面视图，可直接按灰度图处理；访问每行必须使用 `frame->stride`，不能假设 `stride == width`。
  输出：`0` 表示处理成功；默认弱符号实现采用简单帧差，首帧只建立基线，后续统计亮度差超过阈值的像素百分比并写入 `motion_score`，达到阈值时置 `is_motion = 1`。该文件是算法兜底示例；当前构建同时编译 `motion_detect.c`，链接时由 B 侧强符号覆盖它。

### big_core/mpp_pipeline/motion_thread.c

- `ai_motion_thread(arg: void *) -> void`
  输入：RT-Thread 线程参数，当前未使用。
  输出：无线程返回值；循环执行 `osd_poll_auto_hide()`、`ai_frame_try_get()`、`motion_adapter_process()`，随后先用局部变量 `ai_frame_handle` 调用 `ai_frame_release()` 释放 dump/mmap 资源，最后才按事件调用 `osd_set_motion_visible()` 和 `stream_export_request_snapshot()`。`ai_frame_handle` 不参与算法计算，只作为当前帧资源释放凭证。

- `ai_motion_thread_start(void) -> k_s32`
  输入：无。
  输出：`0` 表示 AI frame、motion adapter、RT-Thread 线程启动成功。

- `wait_ai_motion_thread_exit(timeout_ms: k_u32) -> k_bool`
  输入：等待 AI 运动线程退出的超时时间，当前由 `AI_THREAD_EXIT_TIMEOUT_MS` 给出。
  输出：`K_TRUE` 表示收到 `g_ai_motion_exit_sem`；`K_FALSE` 表示等待超时。该函数为文件内静态辅助函数。

- `ai_motion_thread_stop(void) -> void`
  输入：无。

## 2. 调用链

### 初始化链路

```text
main()
-> vb_init()
-> venc_create_chn(VENC_CHN, ENC_BITRATE)
   -> kd_mpi_venc_create_chn()
   -> g_status = STATUS_VENC_CREATED
-> osd_init()
-> venc_start_chn(VENC_CHN)
-> vicap_config(SENSOR_TYPE)
-> vicap_try_config()
-> vi_bind_venc()
-> vicap_start()
-> stream_export_init(STREAM_EXPORT_DATAFIFO 或 STREAM_EXPORT_LOCAL_LOG)
-> ai_motion_thread_start()
-> rt_thread_create(stream_thread)
```

关键数据：

- `vb_init()` 准备主帧、VENC 码流、AI 低清帧三类内存池。
- `venc_create_chn()` 创建 VENC 通道并设置 `STATUS_VENC_CREATED`，但不启用主动 IDR，也不启动编码；CBR 输入/目标帧率为 `30/15fps`，自然 GOP 为 `8`，VENC stream buffer 数量为 `OUTPUT_BUF_CNT=3`。
- `osd_init()` 在 VENC start 前 attach 2D OSD，满足 SDK 对 `kd_mpi_venc_attach_2d()` 的调用顺序要求。
- `venc_start_chn()` 在 OSD attach 和参数下发完成后启动编码通道。
- `vicap_try_config()` 建立主通道 `1920x1080 NV12` 和 AI 通道 `640x480 NV12`，两路通道属性都请求 `VICAP_OUTPUT_FPS=15`。
- `vi_bind_venc()` 只绑定主通道到 VENC。
- `stream_export_init()` 决定后续导出后端：本地日志立即释放，DATAFIFO 等小核释放；它必须早于 `ai_motion_thread_start()`，避免 AI 线程刚检测到运动时快照出口尚未初始化。
- `g_stream_running` 在线程启动前置 `1`；`Ctrl+C` 只关闭 `g_running`，不立即关闭 stream 线程，保证退出清理阶段还能继续消费 VENC buffer。

### 本地日志码流链路

```text
stream_thread()
-> stream_export_flush()
-> nalu_ipc_drain_releases()，独立推进 READ_DONE callback
-> kd_mpi_venc_query_status()
-> kd_mpi_venc_get_stream()
-> pack_cnt == 0: 限频记录 VENC empty stream，5ms 后重试
-> venc_stream_video_pts()
-> stream_freshness_observe()
   -> stale: kd_mpi_venc_release_stream()，继续追帧
   -> fresh: 继续导出
-> stream_export_submit_venc_stream()
-> release_by_caller = K_TRUE
-> kd_mpi_venc_release_stream()
```

关键数据：

- `k_venc_stream` 直接提交给导出层，本地日志后端只读取 pack 的 `len` 和 `type`。
- `stream_export_submit_venc_stream()` 在本地日志后端限频打印 `Get NALU, Size: ...`，配合 `stream_thread()` 每轮主动让出调度，避免串口/终端输出和高优先级采集线程拖慢 Ctrl+C 退出响应。
- 打印完成后立即释放 VENC stream。
- 本地日志模式 `stream_export_get_pending_count()` 始终返回 `0`。

### DATAFIFO 码流链路

```text
stream_thread()
-> stream_export_flush()
-> nalu_ipc_drain_releases()，独立推进 READ_DONE callback
-> kd_mpi_venc_query_status()
-> kd_mpi_venc_get_stream()
-> pack_cnt == 0: 不进入导出层，5ms 后重试
-> stream_export_submit_venc_stream()
-> nalu_ipc_submit_stream(..., flags)
-> nalu_ipc_drain_releases()
-> kd_datafifo_write(mpp_nalu_ipc_msg)
-> 小核 mmap phys_addr 读取 NALU
-> 小核检查 reserved 快照标志并在 READ_DONE 前复制被标记帧
-> 小核 DATAFIFO_CMD_READ_DONE
-> 小核发送前 rtp_get_send_queue_bytes(SIOCOUTQ)
-> 小核 H.265 RTP 分片与 sendto()
-> 小核发送后再次读取 SIOCOUTQ 并输出 rtp:diag
-> stream_export_flush() 或 nalu_ipc_deinit()
-> nalu_ipc_release_callback()
-> kd_mpi_venc_release_stream()
```

关键数据：

- 大核写入 DATAFIFO 的不是 NALU 内容，而是 `mpp_nalu_ipc_msg` 描述符。
- 每条消息包含 `seq`、`frame_pts`、`submit_time_ms`、`total_len` 和 `reserved`；每个 pack 包含 `phys_addr`、`len`、`pts`、`type`。`reserved` 的 bit0 是 `MPP_NALU_IPC_FLAG_SNAPSHOT`，用于通知小核复制该帧码流并投递快照队列。
- DATAFIFO ring entries 保持 `3`，pending 表容量为 `2`，VENC 输出 buffer 数量为 `3`；两个 slot 覆盖 `15fps × 100ms ≈ 1.5` 个实测在途帧，并保留 1 个 VENC buffer 余量。
- `media_venc.c` 不直接调用 `nalu_ipc_*`，只根据导出层返回的 `release_by_caller` 决定是否立即释放。
- 大核不能在小核 `READ_DONE` 前释放 stream。
- 若 DATAFIFO 无可写空间、pending 表满或写入失败，导出层返回错误并要求 `stream_thread()` 立即 release 当前 stream，形成“丢当前帧、保留已有 pending 生命周期”的安全低延迟策略。pending 满时大核会先有界 drain release 通知并重试一次；仍失败才丢当前帧。大核不会仅凭 `avail_write_len` 变大就强行释放仍可能被小核使用的旧 DATAFIFO 描述符。
- 小核 DATAFIFO 读端必须保证 `kd_datafifo_read()` 成功后最终调用 `DATAFIFO_CMD_READ_DONE`；RTSP 发送、网络阻塞、SD 卡写入、JPEG/快照保存不能阻塞在 `READ_DONE` 前。快照帧只允许在 `READ_DONE` 前复制必要编码数据，复制后应立即 `READ_DONE`，后续保存异步处理。
- 如果本次 stream 计划携带快照标志但 DATAFIFO 投递失败，导出层不消费快照请求；下一帧会继续尝试设置 `reserved`，避免快照请求因为拥塞被误确认。
- `nalu_ipc_submit_stream()` 对每次尝试递增 `seq`；小核收到的 `seq` 若跳号，表示大核低延迟模式下发生了主动跳帧。
- `nalu_ipc_release_callback()` 用同一条消息里的 `submit_time_ms` 计算 `read_done_age_ms`；若该值长期低于 `100ms` 且 `pending` 稳定在 `0-1`，大核 DATAFIFO/VENC 路径基本可排除。
- 大核稳定性诊断版在 `nalu_ipc_submit_stream()` 的初始化、drain、空间查询、pending、write和WRITE_DONE失败出口统一打印 `[ipc:submit-fail]`；`nalu_ipc_drain_releases()` 在返回错误或pending无进展时打印 `[ipc:drain]`。`stream_export_submit_venc_stream()` 的早退打印 `[export:fail]`，`stream_thread()` 同时记录 `status_cur_packs/output_pack_cnt/release_by_caller`。零包修复后，`stream_thread()` 每轮在查询 VENC 前主动 flush release 通知，并把成功返回的零 pack 截止在导出层之前，避免 `pending=1 -> VENC 无包 -> 无有效提交可触发 drain` 的自锁。
- `nalu_ipc_log_stats()` 低频输出 `seq/pending/seq_gap/drop_current/read_done_seq/read_done_age_ms/ok/datafifo_full/pending_full/write_fail/release_done`，用于判断积压发生在哪一段；`nalu_ipc_log_pending_slots()` 在 pending 满时输出每个未释放 slot 的年龄，重点排查 `pending_full_after_drain` 且 `avail=1536` 的场景。
- 小核 RTSP 队列应配合保持 `2-3` 帧深度，取队列时 drain 到最新帧；本仓库没有小核 pusher 源码，因此这里仅记录对接约束。

### AI 旁路链路

```text
ai_motion_thread()
-> osd_poll_auto_hide()
-> ai_frame_try_get()
-> motion_adapter_process()
-> motion_detect_process()
-> 连续帧确认 / 锁存 / 冷却消抖
-> ai_frame_release()
-> osd_set_motion_visible()
-> stream_export_request_snapshot()
```

关键数据：

- AI 线程只读取 `AI_VICAP_CHN` 的低清 Y 平面，主通道编码链路不被 CPU 拷贝打断。
- `ai_gray_frame_view.y` 是 AI 低清通道 NV12 的 Y 平面，可直接作为灰度图输入；`stride` 是每行跨度，可能大于 `width`。
- `motion_detect.c` 是当前实际生效的 B 侧强符号算法；`motion_detect_default.c` 继续保留弱符号兜底示例。
- `motion_detect_process()` 只输出 `is_motion` 和 `motion_score`，不直接操作 OSD、VICAP、VENC、线程或帧释放。
- `motion_adapter_process()` 先把算法结果送入消抖状态机；连续 `3` 帧运动后才生成事件，持续运动期间锁存避免每帧触发，连续 `5` 帧静止后释放锁存，且两次事件之间至少间隔 `1000ms`。确认事件同时携带 `request_snapshot = 1` 和 `osd_duration_ms = 1000`。
- OSD 使用 VENC 2D OSD 叠加 ARGB8888 图片；`osd_set_motion_visible()` 只负责显隐和自动隐藏 deadline，不参与运动事件消抖。AI frame 必须先释放，再调用 OSD 显示，避免 OSD/VENC 2D 调用阻塞时耗尽 AI dump buffer。
- 快照请求只进入 `stream_export.c` 的小型 FIFO，不调用主动 IDR API；后续码流处理等待编码器按 GOP 自然产生带 `K_VENC_HEADER` 的 stream，超过 `VENC_GOP + 2` 帧后使用现有兜底选择。大核不写 SD 卡、不复制图片、不做 JPEG 编码，只在后续 DATAFIFO 消息的 `reserved` 里设置快照 bit。

### 清理链路

```text
sig_handler()
-> g_running = 0
-> pipeline_cleanup()
-> Cleanup flow: producer-stop-before-stream-stop
-> ai_motion_thread_stop()
-> 有限等待 g_ai_motion_exit_sem
-> 超时则继续后续硬件清理
-> kd_mpi_vicap_stop_stream()
-> kd_mpi_sys_unbind()
-> kd_mpi_vicap_deinit()
-> stream_thread_stop()
-> g_stream_running = 0
-> 有限等待 g_stream_exit_sem
-> stream_export_deinit()
-> kd_mpi_venc_stop_chn()
-> osd_deinit()
-> kd_mpi_venc_destroy_chn()
-> kd_mpi_venc_close_fd()
-> kd_mpi_vb_exit()
```

关键数据：

- `sig_handler()` 只修改 `sig_atomic_t` 标志，不调用 `LOG()` 或 `printf()`。
- `stream_thread()` 不直接随 `g_running` 退出；cleanup 先停止生产者 VICAP，让 stream 线程继续 drain VENC，再通过 `stream_thread_stop()` 有限等待退出，避免停止 VICAP 时下游 VENC buffer 无人消费。
- AI 线程停止等待有 `AI_THREAD_EXIT_TIMEOUT_MS` 上限，避免 cleanup 卡死。
- OSD 自动隐藏由 AI 线程轮询 `osd_poll_auto_hide()` 完成，不再使用后台隐藏线程、RT timer 或额外显隐锁；退出时 `osd_deinit()` 清掉 pending deadline。
- DATAFIFO 模式下，`stream_export_deinit()` 会调用 `nalu_ipc_deinit()`。
- `stream_export_deinit()` 会丢弃尚未投递的快照请求；`nalu_ipc_deinit()` 会强制释放尚未被小核 `READ_DONE` 的 pending stream。
- VENC 2D OSD 在 `venc_create_chn()` 后、`venc_start_chn()` 前 attach；退出时放在 `kd_mpi_venc_stop_chn()` 后、`kd_mpi_venc_destroy_chn()` 前 detach，避免通道仍在运行或绑定时卡在 `kd_mpi_venc_detach_2d()`。

## 3. 数据追踪

典型输入：GC2093 输出一帧 `1920x1080` 图像，同时大核配置 AI 旁路输出 `640x480`。

1. `main()` 初始化 VB、创建 VENC、初始化并 attach OSD、启动 VENC、初始化 VICAP、绑定、启动 VICAP、初始化导出层，最后启动 AI 线程和 stream 线程。VENC 不启用运行期主动 IDR，运动事件也不调用主动 IDR 请求接口。
2. VICAP 主通道属性请求 `VICAP_OUTPUT_FPS=15`，并通过 `kd_mpi_sys_bind()` 自动送入 VENC；板端 AI 帧计数显示硬件实际输入仍可能约 `30fps`，因此 VENC 保留 `SRC_FPS=30`、`DST_FPS=15` 做输出降帧，CPU 不拷贝 1080P 原始帧。
3. `stream_thread()` 每轮先调用 `stream_export_flush()` 回收已由小核 `READ_DONE` 的 pending stream，再查询并调用 `kd_mpi_venc_get_stream()`。成功但 `pack_cnt=0` 的返回只记录空流诊断并重试；有效 H.265 pack 才从非 Header PTS 计算相对首帧的 `source_drift_ms`。
4. 连续两帧漂移超过 `70ms` 时直接释放迟到 stream；漂移恢复到 `35ms` 后，`stream_thread()` 才继续把新鲜 `k_venc_stream` 提交给 `stream_export_submit_venc_stream()`。
5. 如果是 `STREAM_EXPORT_LOCAL_LOG`，pack 长度被打印到终端，导出层返回 `release_by_caller = K_TRUE`，大核立即调用 `kd_mpi_venc_release_stream()`。
6. 如果是 `STREAM_EXPORT_DATAFIFO`，导出层调用 `nalu_ipc_submit_stream(..., flags)`；函数先分配新的 `seq` 和 `submit_time_ms`，再把 pack 和 `reserved` 标志整理为 `mpp_nalu_ipc_msg` 写入 DATAFIFO，小核按 `phys_addr + len` mmap 并发送 RTSP/RTP。
7. DATAFIFO 成功接管时导出层返回 `release_by_caller = K_FALSE`，`stream_thread()` 不立即释放该 stream，pending 深度加 1。
8. 如果 DATAFIFO 或 pending 已满，本次 `seq` 被计入 drop，导出层返回错误并要求 `stream_thread()` 立即释放当前 stream。小核下一次收到更大的 `seq` 时可用跳号判断大核已丢帧。
9. 小核 `datafifo_rtp_sender_loop()` 通过 `nalu_datafifo_read()` 取出描述符，`datafifo_copy_msg_packs()` 把需要发送或保存的 H.265 pack 复制到小核自有内存，随后立即通过 `nalu_datafifo_read_done()` 调用 `DATAFIFO_CMD_READ_DONE`。大核 release callback 用同一个 `seq` 计算 `read_done_age_ms`，再重建 `k_venc_stream` 并释放 VENC buffer，pending 深度减 1。
10. READ_DONE 完成后，小核才调用 `datafifo_process_copied_frame_for_playback()`，沿 `datafifo_process_h265_buffer_for_playback() -> datafifo_process_nalu_for_playback() -> send_h265_nalu_rtp() -> send_rtp_packet()` 解析 Annex-B、执行 H.265 FU 分片并发送 RTP。UDP socket 由 `create_udp_socket_only()` 配置 `O_NONBLOCK`、`SO_SNDBUF=128KiB` 和低延迟 TOS；发送队列繁忙时丢当前包并等待下一次自然随机访问帧，不能阻塞下一轮 DATAFIFO 读取。`rtp_clock_next_timestamp()` 位于独立 `rtp_clock` 模块：首帧 PTS 为 0 时先按固定步长推进，下一帧首次获得有效 PTS 后从 `last_rtp_ts + RTP_TS_STEP` 建立新的 PTS/RTP 基准；PTS 回退或单次跳变超过 `1s` 时，异常帧只推进一个固定 step，同时把该 PTS 设为新基线，下一帧恢复 `pts_mode=1`，避免永久固定步长导致 RTP 时间轴落后墙钟。`datafifo_rtp_sender_loop()` 在重建基线时输出一次 `[rtp:rebase] count/from_pts/to_pts/rtp_ts`。诊断逻辑还通过 `SIOCOUTQ`、`rtp_diag_log_frame()` 和 `datafifo_log_health()` 对齐 UDP 队列、RTP/墙钟漂移及 READ_DONE 进度，大于64KiB帧通过四阶段检查点追踪。
11. AI 旁路线程并行从同样请求 15fps 的低清通道 dump 帧，把 NV12 的 Y 平面填成 `ai_gray_frame_view` 传给运动检测适配层；算法按 `stride` 逐行读取有效 `width` 字节。
12. `motion_detect.c` 的 B 侧强符号帧差算法首帧建立上一帧缓存，后续抽样比较 Y 分量，输出千分比 `motion_score` 和 `is_motion`；`motion_detect_default.c` 的弱符号默认算法仍保留为兜底示例。
13. `motion_adapter_process()` 对算法输出做消抖：连续 `3` 帧运动才生成事件，持续运动期间锁存，连续 `5` 帧静止后释放锁存，短时间重复运动受 `1000ms` 冷却限制。
14. 确认运动事件产生后，`ai_motion_thread()` 先释放当前 AI frame，再打印 `Motion detected: event_id=... score=...`，调用 `osd_set_motion_visible(1, 1000)`，并通过 `stream_export_request_snapshot()` 登记一次快照请求。请求成功入队后直接返回，不触发主动 IDR。
15. `stream_export_submit_venc_stream()` 发现待快照请求后，优先等待带 `K_VENC_HEADER` 的后续 stream；等到目标 stream 或超过等待帧数后，把 `MPP_NALU_IPC_FLAG_SNAPSHOT` 写入 `reserved` 并在 DATAFIFO 投递成功后消费请求。
16. 小核 DATAFIFO/RTP 线程读取到 `reserved & MPP_NALU_IPC_FLAG_SNAPSHOT` 后，必须在 `DATAFIFO_CMD_READ_DONE` 前复制被标记帧的编码数据，再投递给小核快照队列；大核仍只等待 READ_DONE 后释放 VENC buffer。
17. 退出时，主线程不先等待 stream 线程；`pipeline_cleanup()` 先停止 AI 线程，再停止 VICAP 生产者，让 stream 线程继续 drain VENC。
18. VICAP stop/unbind/deinit 后，`stream_thread_stop()` 设置 `g_stream_running = 0` 并有限等待 stream 线程退出，再释放导出层、VENC、OSD buffer 和 VB。
19. 主循环当前只等待 `SIGINT`/`SIGTERM` 让 `sig_handler()` 清零 `g_running`；原 `AUTO_EXIT_SEC=600` 自动退出验收逻辑保留在源码禁用块中，默认不会在 10 分钟后退出。

最终输出：

- 本地日志模式：终端限频打印 NALU 大小和统计信息。
- DATAFIFO 模式：终端打印 DATAFIFO 物理地址，小核通过该地址读取 NALU 描述并推流。
- 两种模式默认持续运行，收到退出信号后应打印 `Cleanup complete` 和 `Pipeline test PASSED.`。

### 码流描述的三层关系

```text
k_venc_stream / k_venc_pack
  SDK get_stream()/release_stream 使用；packs[] 是大核线程栈上的临时描述数组。

mpp_nalu_ipc_msg
  跨核 DATAFIFO 消息；保存 seq、frame_pts、submit_time_ms、total_len、reserved 以及每个 pack 的 phys_addr、len、pts、type 等小核消费、快照识别和延迟测量所需字段。

g_pending[].ipc_msg
  大核 DATAFIFO 后端的延迟释放凭据；小核 READ_DONE 后用它重建 k_venc_stream 并 release。
```

真实 H.265 数据只在 VENC buffer 中，不会复制到 DATAFIFO 或 pending 表；多份结构复制的是描述信息，用于跨越 SDK、跨核和生命周期三个边界。

## 4. 日志策略

默认运行日志只保留有状态判断价值的内容：

- 保留：模块初始化成功/失败、传感器识别结果、VI->VENC 绑定、VICAP 启动、导出模式、DATAFIFO `phy_addr`、运行中统计、运动事件、退出清理和 PASS/FAIL。
- 保留：所有 SDK/API 失败返回码，以及 DATAFIFO full、pending full、drop current、pending 强制释放等会影响资源释放或跨核对接的异常状态。
- 保留：VENC/IPC 启动配置指纹；`stream_thread()` 的 `pending/source_drift_ms/max_drift_ms/stale_drop`；`nalu_ipc_log_stats()` 的 `pending_high_water/drop/full/seq gap/read done age`；pending slot 年龄和慢 `READ_DONE` 诊断。
- 小核 queueprobe 诊断版保留：socket 请求/实际 `SO_SNDBUF`，以及 `[rtp:diag]` 的 `outq_before/outq_after/outq_high/rtp_step/clock_drift_ms/query_fail`；前 5 帧强制打印，之后每 15 帧或高水位增长、时间戳回退、查询失败时打印。
- RTP PTS 不连续时保留一次 `[rtp:rebase] count/from_pts/to_pts/rtp_ts`；异常帧保持 RTP 单调，下一帧应恢复 `pts_mode=1`，用于确认时钟没有永久退回固定 `6000` 步长。
- 稳定性专项诊断保留：大核 `[ipc:submit-fail]/[ipc:drain]/[export:fail]` 和小核 `[datafifo:health]/[datafifo:large]`；正常高频路径限频，首次失败、每15次重复失败和大帧强制输出。
- 保留：快照请求入队、队列满丢弃、DATAFIFO 成功携带快照标志、本地日志模式模拟快照事件，以及 `[big] snapshot flag set seq=... pts=... reserved=0x1`，用于和小核 `[datafifo] flags=0x1`、`[snapshot] queued/saved` 日志对齐。
- 省略：VICAP channel format、线程 timeout/max_packs、AI 首帧指针/stride 等不会帮助区分积压边界的纯系统参数。
- 若需要排查参数配置，优先查源码宏和板端专项调试日志，不在默认稳定性运行日志里长期打印。
