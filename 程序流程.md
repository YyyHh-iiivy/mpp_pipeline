# 程序流程

本文用于同步记录当前 `big_core/mpp_pipeline` 的代码结构、调用关系和数据流。每次新增或修改代码文件后都应同步更新。

文件命名前缀按任务目标划分：

- `media_*`：大核 MPP 底层媒体能力，包括 VB、VICAP、VENC、硬件绑定、OSD 和资源清理。
- `stream_*`：H.265/NALU 码流出口和小核 RTSP/RTP 对接准备。
- `motion_*`：AI 低清帧、运动检测算法契约和运动事件闭环。
- `mpp_pipeline.*`、`mpp_types.h`、`mpp_nalu_ipc.h` 保持为总入口和公共契约。

## 1. 函数抽象

### big_core/mpp_pipeline/mpp_types.h

- `AI_GRAY_MAX_WIDTH / AI_GRAY_MAX_HEIGHT -> compile_time_contract`
  输入：无运行期输入。
  输出：AI 灰度检测输入的最大宽高，当前为 `640x480`；`mpp_pipeline.h` 的 `AI_WIDTH`、`AI_HEIGHT` 与其保持一致。

- `ai_gray_frame_view(frame_id, timestamp_ms, width, height, stride, y) -> ai_luma_view`
  输入：AI 低清通道 dump 出来的 NV12 Y 平面元数据。
  输出：运动检测只读灰度视图；`y` 指向当前进程可读的 Y 平面，调用方不转移所有权，处理完仍由 `ai_frame_release()` 释放。

- `motion_detect_result(is_motion, motion_score) -> detection_result`
  输入：运动检测算法填充。
  输出：`is_motion != 0` 表示检测到运动；`motion_score` 为算法给出的强度分数，默认实现使用变化像素百分比。

### big_core/mpp_pipeline/motion_detect.h

- `motion_detect_process(frame: const ai_gray_frame_view *, result: motion_detect_result *) -> k_s32`
  输入：AI 灰度帧和检测结果输出结构。
  输出：`0` 表示处理成功；队员 B 可以提供同名强符号实现覆盖默认弱符号示例。B 侧实现只负责检测并填写 `result`，不能直接操作 OSD、VICAP、VENC、RT-Thread 线程或释放帧资源。

### big_core/mpp_pipeline/mpp_pipeline.c

- `main(void) -> int`
  输入：无；读取编译期宏配置，如 `SENSOR_TYPE`、`ENC_WIDTH`、`ENC_HEIGHT`、`STREAM_EXPORT_DATAFIFO`。
  输出：程序退出码；`0` 表示初始化、运行、清理完成，非 `0` 表示失败。

- `sig_handler(signo: int) -> void`
  输入：`SIGINT` 或 `SIGTERM` 等信号编号。
  输出：无；只设置 `g_stop_signal` 和 `g_running = 0`，触发主线程、码流线程和 AI 线程退出。该函数不打印日志，避免在信号上下文里调用 `printf` 导致 Ctrl+C 退出卡住。

- `wait_stream_thread_exit(timeout_ms: k_u32) -> k_bool`
  输入：等待 VENC 码流线程退出的超时时间，当前由 `STREAM_THREAD_EXIT_TIMEOUT_MS` 给出。
  输出：`K_TRUE` 表示收到 `g_stream_exit_sem`；`K_FALSE` 表示等待超时。该函数为文件内静态辅助函数。

### big_core/mpp_pipeline/media_vb.c

- `vb_init(void) -> k_s32`
  输入：无；使用 `FRAME_BUF_SIZE`、`STREAM_BUF_SIZE`、`AI_CHN_BUF_SIZE` 和各 pool 块数宏。
  输出：`0` 表示 VB pool 初始化成功；非 `0` 表示 SDK 调用失败。

### big_core/mpp_pipeline/media_venc.c

- `venc_init(chn: k_u32, bitrate: k_u32) -> k_s32`
  输入：VENC 通道号、目标码率 kbps。
  输出：`0` 表示 H.265 CBR 编码通道创建并启动成功；非 `0` 表示失败。

- `stream_thread(arg: void *) -> void`
  输入：RT-Thread 线程参数，转换为 VENC 通道号。
  输出：无；循环获取 VENC stream、统计非 header 帧数和字节数，并把 stream 交给导出层。是否立即释放由 `stream_export_submit_venc_stream()` 返回的 `release_by_caller` 决定。

### big_core/mpp_pipeline/media_vicap.c

- `vicap_try_config(sensor_type: k_vicap_sensor_type) -> k_s32`
  输入：传感器类型。
  输出：`0` 表示 sensor 查询、device 配置、主通道和 AI 通道配置、VICAP 初始化成功。

- `vicap_config(sensor_type: k_vicap_sensor_type) -> k_s32`
  输入：首选传感器类型。
  输出：`0` 表示首选或 fallback 配置成功；非 `0` 表示全部失败。

- `vicap_start(void) -> k_s32`
  输入：无。
  输出：`0` 表示 VICAP stream 启动成功。

### big_core/mpp_pipeline/media_bind.c

- `vi_bind_venc(void) -> k_s32`
  输入：无；使用固定 `VICAP_CHN` 和 `VENC_CHN`。
  输出：`0` 表示主通道 `VI -> VENC` 硬件绑定成功。

### big_core/mpp_pipeline/stream_export.c

- `stream_export_init(mode: stream_export_mode) -> k_s32`
  输入：导出模式，当前常用 `STREAM_EXPORT_DATAFIFO` 或 `STREAM_EXPORT_LOCAL_LOG`。
  输出：`0` 表示导出层初始化成功；DATAFIFO 模式会同时初始化 `nalu_ipc_module`。

- `stream_export_submit_venc_stream(chn: k_u32, stream: const k_venc_stream *, release_by_caller: k_bool *) -> k_s32`
  输入：VENC 通道号、`kd_mpi_venc_get_stream()` 得到的 stream、释放责任输出指针。
  输出：`0` 表示导出层处理成功；本地日志模式打印 NALU 并设置 `release_by_caller = K_TRUE`，DATAFIFO 成功接管时设置 `release_by_caller = K_FALSE`，DATAFIFO 投递失败时返回错误并要求调用方立即释放。

- `stream_export_flush(void) -> k_s32`
  输入：无。
  输出：DATAFIFO 模式下触发 `nalu_ipc_flush()` 处理小核 `READ_DONE` 回调；本地日志模式直接返回 `0`。

- `stream_export_deinit(void) -> void`
  输入：无。
  输出：无；DATAFIFO 模式下会反初始化 `nalu_ipc_module`。

### big_core/mpp_pipeline/stream_nalu_ipc.c

- `nalu_ipc_init(void) -> k_s32`
  输入：无；使用 DATAFIFO writer 参数 `entries=OUTPUT_BUF_CNT`、`item_size=512`。
  输出：`0` 表示 DATAFIFO writer 打开成功，并打印给小核使用的 `phy_addr`。

- `nalu_ipc_submit_stream(chn: k_u32, stream: const k_venc_stream *) -> k_s32`
  输入：VENC 通道号和 `kd_mpi_venc_get_stream()` 得到的 stream。
  输出：`0` 表示已把 NALU 描述符写入 DATAFIFO；非 `0` 表示未投递成功，导出层会要求 `media_venc.c` 立即 release 当前 stream。

- `nalu_ipc_flush(void) -> k_s32`
  输入：无。
  输出：DATAFIFO `write(NULL)` 的结果；用于处理小核 `READ_DONE` 后触发的释放回调。

- `nalu_ipc_get_phy_addr(void) -> k_u64`
  输入：无。
  输出：DATAFIFO ring buffer 物理地址，供小核 `kd_datafifo_open_by_addr()` 使用。

- `nalu_ipc_deinit(void) -> void`
  输入：无。
  输出：无；flush DATAFIFO，强制释放未被小核 `READ_DONE` 的 pending stream，并关闭 DATAFIFO。

### big_core/mpp_pipeline/motion_ai_frame.c

- `ai_frame_channel_init(void) -> k_s32`
  输入：无。
  输出：`0` 表示 AI 帧读取逻辑初始化完成。

- `ai_frame_try_get(view: ai_gray_frame_view *, handle: void **) -> k_s32`
  输入：输出视图和资源句柄指针。
  输出：`0` 表示从 AI VICAP 通道 dump 到一帧并 mmap Y 平面。

- `ai_frame_release(handle: void *) -> k_s32`
  输入：`ai_frame_try_get()` 返回的资源句柄。
  输出：`0` 表示 munmap 并释放 dump frame 成功。

- `ai_frame_channel_deinit(void) -> void`
  输入：无。
  输出：无；输出反初始化日志。

### big_core/mpp_pipeline/motion_adapter.c

- `motion_adapter_init(void) -> k_s32`
  输入：无。
  输出：`0` 表示运动适配层初始化完成，并重置事件序号。

- `motion_adapter_process(frame: const ai_gray_frame_view *, event: motion_event_msg *, has_event: k_bool *) -> k_s32`
  输入：AI 灰度帧、事件输出、是否有事件输出标志。
  输出：`0` 表示处理成功；内部调用 `motion_detect_process()`，检测到运动时填充 `motion_event_msg`。

- `motion_adapter_deinit(void) -> void`
  输入：无。
  输出：无；输出运动适配层反初始化日志。

### big_core/mpp_pipeline/motion_detect_default.c

- `motion_validate_frame(frame: const ai_gray_frame_view *) -> k_s32`
  输入：AI 灰度帧视图。
  输出：`0` 表示 `frame`、`y`、宽高、`stride` 和最大尺寸都满足默认算法要求；非 `0` 表示非法输入。该函数为文件内静态辅助函数。

- `motion_copy_baseline(frame: const ai_gray_frame_view *) -> void`
  输入：AI 灰度帧视图。
  输出：无；按 `stride` 逐行复制有效 `width` 字节到内部上一帧基线，不复制每行 padding。该函数为文件内静态辅助函数。

- `motion_detect_process(frame: const ai_gray_frame_view *, result: motion_detect_result *) -> k_s32`
  输入：AI 低清通道 NV12 的 Y 平面视图，可直接按灰度图处理；访问每行必须使用 `frame->stride`，不能假设 `stride == width`。
  输出：`0` 表示处理成功；默认弱符号实现采用简单帧差，首帧只建立基线，后续统计亮度差超过阈值的像素比例并写入 `motion_score`，达到阈值时置 `is_motion = 1`。

### big_core/mpp_pipeline/motion_thread.c

- `ai_motion_thread_start(void) -> k_s32`
  输入：无。
  输出：`0` 表示 AI frame、motion adapter、RT-Thread 线程启动成功。

- `wait_ai_motion_thread_exit(timeout_ms: k_u32) -> k_bool`
  输入：等待 AI 运动线程退出的超时时间，当前由 `AI_THREAD_EXIT_TIMEOUT_MS` 给出。
  输出：`K_TRUE` 表示收到 `g_ai_motion_exit_sem`；`K_FALSE` 表示等待超时。该函数为文件内静态辅助函数。

- `ai_motion_thread_stop(void) -> void`
  输入：无。
  输出：无；停止 AI 线程，有限等待退出信号量；若线程仍未退出，则记录日志并继续后续硬件清理，避免 cleanup 永久阻塞。

### big_core/mpp_pipeline/media_osd.c

- `osd_init(void) -> k_s32`
  输入：无。
  输出：`0`；当前为 OSD stub 初始化。

- `osd_set_motion_visible(visible: k_u32, duration_ms: k_u32) -> k_s32`
  输入：显示标志和持续时间。
  输出：`0`；当前只打印 stub 日志。

- `osd_deinit(void) -> void`
  输入：无。
  输出：无；反初始化 OSD stub。

### big_core/mpp_pipeline/media_cleanup.c

- `pipeline_cleanup(void) -> void`
  输入：无；读取 `g_status` 判断资源初始化进度。
  输出：无；逆序停止 AI、导出层、OSD、VICAP、绑定、VENC 和 VB。

## 2. 调用链

### 初始化链路

```text
main()
-> vb_init()
-> venc_init(VENC_CHN, ENC_BITRATE)
-> osd_init()
-> vicap_config(SENSOR_TYPE)
-> vicap_try_config()
-> vi_bind_venc()
-> vicap_start()
-> ai_motion_thread_start()
-> stream_export_init(STREAM_EXPORT_DATAFIFO 或 STREAM_EXPORT_LOCAL_LOG)
-> rt_thread_create(stream_thread)
```

关键数据：

- `vb_init()` 准备主帧、VENC 码流、AI 低清帧三类内存池。
- `vicap_try_config()` 建立主通道 `1920x1080 NV12` 和 AI 通道 `640x480 NV12`。
- `vi_bind_venc()` 只绑定主通道到 VENC。
- `stream_export_init()` 决定后续导出后端：本地日志立即释放，DATAFIFO 等小核释放。

### 本地日志码流链路

```text
stream_thread()
-> kd_mpi_venc_query_status()
-> kd_mpi_venc_get_stream()
-> stream_export_submit_venc_stream()
-> release_by_caller = K_TRUE
-> kd_mpi_venc_release_stream()
```

关键数据：

- `k_venc_stream` 直接提交给导出层，本地日志后端只读取 pack 的 `len` 和 `type`。
- `stream_export_submit_venc_stream()` 在本地日志后端打印 `Get NALU, Size: ...`。
- 打印完成后立即释放 VENC stream。

### DATAFIFO 码流链路

```text
stream_thread()
-> kd_mpi_venc_query_status()
-> kd_mpi_venc_get_stream()
-> stream_export_submit_venc_stream()
-> nalu_ipc_submit_stream()
-> kd_datafifo_write(mpp_nalu_ipc_msg)
-> 小核 mmap phys_addr 读取 NALU
-> 小核 DATAFIFO_CMD_READ_DONE
-> stream_export_flush() 或 nalu_ipc_deinit()
-> nalu_ipc_release_callback()
-> kd_mpi_venc_release_stream()
```

关键数据：

- 大核写入 DATAFIFO 的不是 NALU 内容，而是 `mpp_nalu_ipc_msg` 描述符。
- 每个 pack 包含 `phys_addr`、`len`、`pts`、`type`。
- DATAFIFO queue 深度、pending 表容量和 VENC 输出 buffer 数量对齐，最多跟踪 `OUTPUT_BUF_CNT` 个已投递但未 `READ_DONE` 的 stream。
- `media_venc.c` 不直接调用 `nalu_ipc_*`，只根据导出层返回的 `release_by_caller` 决定是否立即释放。
- 大核不能在小核 `READ_DONE` 前释放 stream。
- 若 DATAFIFO 后端提交失败，导出层返回错误并要求 `stream_thread()` 立即 release 当前 stream，避免 VENC buffer 泄漏。

### AI 旁路链路

```text
ai_motion_thread()
-> ai_frame_try_get()
-> motion_adapter_process()
-> motion_detect_process()
-> osd_set_motion_visible()
-> ai_frame_release()
```

关键数据：

- AI 线程只读取 `AI_VICAP_CHN` 的低清 Y 平面，主通道编码链路不被 CPU 拷贝打断。
- `ai_gray_frame_view.y` 是 AI 低清通道 NV12 的 Y 平面，可直接作为灰度图输入；`stride` 是每行跨度，可能大于 `width`。
- `motion_adapter_process()` 只负责调用算法并把 `motion_detect_result` 包装成 `motion_event_msg`。
- 当前默认弱符号算法在 `motion_detect_default.c` 中提供简单帧差示例，能输出 `is_motion` 和 `motion_score` 用于验证事件链路。
- 队员 B 后续只需提供同名强符号 `motion_detect_process(frame, result)` 覆盖默认实现；B 侧不直接操作 OSD、VICAP、VENC、线程或帧释放。
- OSD 当前是 stub，只打印日志。

### 清理链路

```text
sig_handler() 或 AUTO_EXIT_SEC
-> g_running = 0
-> stream_thread() 退出
-> main() 有限等待 g_stream_exit_sem
-> 超时则继续 cleanup 以打断底层 SDK 阻塞
-> pipeline_cleanup()
-> ai_motion_thread_stop()
-> 有限等待 g_ai_motion_exit_sem
-> 超时则继续后续硬件清理
-> stream_export_deinit()
-> osd_deinit()
-> kd_mpi_vicap_stop_stream()
-> kd_mpi_sys_unbind()
-> kd_mpi_vicap_deinit()
-> kd_mpi_venc_stop_chn()
-> kd_mpi_venc_destroy_chn()
-> kd_mpi_venc_close_fd()
-> kd_mpi_vb_exit()
```

关键数据：

- `sig_handler()` 只修改 `sig_atomic_t` 标志，不调用 `LOG()` 或 `printf()`。
- 主线程等待码流线程退出有 `STREAM_THREAD_EXIT_TIMEOUT_MS` 上限，避免 Ctrl+C 后卡在 `rt_sem_take(..., RT_WAITING_FOREVER)`；超时后继续 cleanup，让 VENC/VICAP stop/deinit 打断底层等待。
- AI 线程停止等待有 `AI_THREAD_EXIT_TIMEOUT_MS` 上限，避免 cleanup 卡死。
- DATAFIFO 模式下，`stream_export_deinit()` 会调用 `nalu_ipc_deinit()`。
- `nalu_ipc_deinit()` 会强制释放尚未被小核 `READ_DONE` 的 pending stream。

## 3. 数据追踪

典型输入：GC2093 输出一帧 `1920x1080` 图像，同时大核配置 AI 旁路输出 `640x480`。

1. `main()` 初始化 VB、VENC、OSD、VICAP、绑定、AI 线程和导出层。
2. 主通道帧通过 `kd_mpi_sys_bind()` 从 VICAP 自动送入 VENC，CPU 不拷贝 1080P 原始帧。
3. `stream_thread()` 调用 `kd_mpi_venc_get_stream()` 取得 H.265 pack。
4. `stream_thread()` 把 `k_venc_stream` 提交给 `stream_export_submit_venc_stream()`。
5. 如果是 `STREAM_EXPORT_LOCAL_LOG`，pack 长度被打印到终端，导出层返回 `release_by_caller = K_TRUE`，大核立即调用 `kd_mpi_venc_release_stream()`。
6. 如果是 `STREAM_EXPORT_DATAFIFO`，导出层调用 `nalu_ipc_submit_stream()`，pack 被整理为 `mpp_nalu_ipc_msg` 写入 DATAFIFO，小核按 `phys_addr + len` mmap 并发送 RTSP/RTP。
7. DATAFIFO 成功接管时导出层返回 `release_by_caller = K_FALSE`，`stream_thread()` 不立即释放该 stream。
8. 小核发送完成后调用 `DATAFIFO_CMD_READ_DONE`，大核 release callback 重建 `k_venc_stream` 并释放 VENC buffer。
9. AI 旁路线程并行 dump 低清帧，把 NV12 的 Y 平面填成 `ai_gray_frame_view` 传给运动检测适配层；算法按 `stride` 逐行读取有效 `width` 字节。
10. `motion_detect_default.c` 的默认帧差算法首帧建立基线，后续输出 `motion_score` 和 `is_motion`；若 B 侧强符号实现存在，链接时覆盖该默认实现。
11. 检测到运动时，`motion_adapter_process()` 生成事件，`ai_motion_thread()` 打印 `Motion detected: event_id=... score=...` 并调用 `osd_set_motion_visible(1, 1000)`。
12. 退出时，主线程有限等待 stream 线程结束；若线程没有及时释放退出信号量，则进入 `pipeline_cleanup()` 继续停止硬件资源，避免 Ctrl+C 永久卡住。
13. `pipeline_cleanup()` 调用 `ai_motion_thread_stop()` 有限等待 AI 线程退出，随后逆序释放所有硬件资源。

最终输出：

- 本地日志模式：终端持续打印 NALU 大小和统计信息。
- DATAFIFO 模式：终端打印 DATAFIFO 物理地址，小核通过该地址读取 NALU 描述并推流。
- 两种模式都应在退出时打印 `Cleanup complete` 和 `Pipeline test PASSED.`。

### 码流描述的三层关系

```text
k_venc_stream / k_venc_pack
  SDK get_stream()/release_stream 使用；packs[] 是大核线程栈上的临时描述数组。

mpp_nalu_ipc_msg
  跨核 DATAFIFO 消息；只保存 phys_addr、len、pts、type 等小核消费所需字段。

g_pending[].ipc_msg
  大核 DATAFIFO 后端的延迟释放凭据；小核 READ_DONE 后用它重建 k_venc_stream 并 release。
```

真实 H.265 数据只在 VENC buffer 中，不会复制到 DATAFIFO 或 pending 表；多份结构复制的是描述信息，用于跨越 SDK、跨核和生命周期三个边界。

## 4. 日志策略

默认运行日志只保留有状态判断价值的内容：

- 保留：模块初始化成功/失败、传感器识别结果、VI->VENC 绑定、VICAP 启动、导出模式、DATAFIFO `phy_addr`、运行中统计、运动事件、退出清理和 PASS/FAIL。
- 保留：所有 SDK/API 失败返回码，以及 DATAFIFO full、pending 强制释放等会影响资源释放或跨核对接的异常状态。
